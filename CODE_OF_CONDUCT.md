# Code of Conduct · 급여납치 행동강령

문서 상태: 최종본
파일 위치: `CODE_OF_CONDUCT.md`
프로젝트: 급여납치 Salary Hijacking Platform
적용 범위: 개발자, 운영자, 관리자, QA, 디자이너, 기획자, 보안 검토자, 외부 기여자, 파트너 협업자
핵심 원칙: 사용자 보호, 금융 데이터 보호, 안전한 협업, 차별 금지, 커뮤니티 신뢰, 보안 우선, 감사 가능성

---

## 1. 목적

급여납치 플랫폼은 사용자의 급여, 고정지출, 고정저축, 일일예산, 변동지출, 납치금액, 알림, 자기계발 활동, 커뮤니티 활동, 광고/제휴 상호작용, 관리자 운영 데이터를 다루는 서비스다.

이 행동강령은 프로젝트 참여자가 개발, 문서화, QA, 운영, 보안 대응, 커뮤니티 관리, 출시, 파트너 협업 과정에서 지켜야 할 행동 기준을 정의한다.

급여납치 프로젝트의 모든 협업은 다음 목표를 우선한다.

1. 사용자의 급여·소비·저축·대출·예산 정보를 안전하게 보호한다.
2. 직장인과 아르바이트생 사용자가 안심하고 자기관리와 금융관리를 할 수 있는 환경을 만든다.
3. 개발·운영·커뮤니티 협업에서 차별, 비방, 괴롭힘, 불법 정보, 보안 위협을 허용하지 않는다.
4. 모든 변경은 서버 권위 계산, RLS, RBAC, 감사로그, 개인정보 보호, 광고 데이터 분리 원칙을 해치지 않는다.
5. 프로젝트의 상용화 품질, 운영 안정성, 사용자 신뢰를 최우선으로 한다.

---

## 2. 적용 대상

이 행동강령은 다음 모든 활동에 적용된다.

- GitHub issue, discussion, pull request, review, commit, release note
- 문서 작성, 기능 명세, UX/UI 설계, API 설계, DB 설계, QA 문서
- mobile 앱, admin 콘솔, API, scheduler, notifications, packages, database, infra 작업
- CI/CD, 배포, rollback, incident response, 보안 스캔, 취약점 대응
- 커뮤니티 모더레이션, CS, 공지, 이벤트, 광고/제휴 운영
- 오프라인 회의, 온라인 회의, 채팅, 이메일, 프로젝트 관리 도구
- 외부 협력사, 보안 점검자, QA 참여자, 운영 대행자와의 협업

프로젝트 공간 밖에서 일어난 행동이라도 급여납치 프로젝트, 사용자, 팀원, 기여자, 파트너, 서비스 신뢰에 영향을 주는 경우 이 행동강령을 적용할 수 있다.

---

## 3. 핵심 가치

### 3.1 사용자 보호

급여납치 사용자는 자신의 월급, 소비, 저축, 예산, 자기관리 루틴을 서비스에 맡긴다. 모든 참여자는 사용자의 민감한 경제생활 정보를 다룬다는 책임감을 가져야 한다.

### 3.2 안전한 협업

협업 과정에서 상대방의 역할, 경력, 성별, 나이, 국적, 장애, 학력, 고용 형태, 직군, 종교, 성적 지향, 정치적 견해, 경제적 상황을 이유로 차별하거나 조롱하지 않는다.

### 3.3 보안 우선

빠른 출시보다 사용자 데이터 보호와 계산 무결성이 우선이다. 인증, 권한, RLS, RBAC, 감사로그, secret 관리, 광고 데이터 분리, 개인정보 보호 정책을 우회하지 않는다.

### 3.4 투명한 책임

결정, 변경, 장애, 보안 이슈, 계산 오류, 데이터 정합성 문제는 숨기지 않는다. 확인 가능한 기록과 재현 가능한 근거를 남긴다.

### 3.5 상용화 품질

모든 코드는 사용자에게 노출될 수 있는 실제 제품 품질을 기준으로 작성한다. 임시 구현, 근거 없는 완성 주장, 테스트 없는 위험 변경, 운영 정책과 충돌하는 변경은 허용하지 않는다.

## 4. 권장 행동

급여납치 프로젝트 참여자는 다음 행동을 실천한다.

1. 논의할 때 사람을 공격하지 않고 문제, 요구사항, 근거, 영향 범위에 집중한다.
2. 기능 변경 시 사용자 가치, 데이터 영향, 보안 영향, 운영 영향, QA 영향, rollback 가능성을 함께 설명한다.
3. 급여·예산·지출·저축 계산 로직은 서버 권위 원칙을 기준으로 검토한다.
4. 민감정보가 포함될 수 있는 로그, 스크린샷, issue, PR, 테스트 fixture를 공유하기 전에 반드시 마스킹한다.
5. 커뮤니티 기능과 관리자 기능은 사용자 안전, 신고 처리, 익명성, 모더레이션 기준을 고려해 설계한다.
6. 광고/제휴 기능은 재무 원천 데이터와 분리된 구조로 설계하고, 동의·정책·감사 가능성을 확인한다.
7. 리뷰에서는 명확하고 실행 가능한 피드백을 제공한다.
8. 모르는 부분은 추측으로 확정하지 않고 관련 문서, 코드, 테스트, 로그, 정책을 확인한다.
9. 장애나 보안 이슈가 발견되면 공개 채널에 세부 악용 방법을 올리지 않고 정해진 보안 절차로 보고한다.
10. 새 기여자가 프로젝트 구조, 문서, 실행 명령, 보안 기준을 이해할 수 있도록 돕는다.
11. 문서와 코드가 충돌하면 충돌 사실을 기록하고 함께 수정한다.
12. 릴리즈 전에는 기능, API, DB, 보안, 개인정보, QA, 운영, rollback 기준을 확인한다.

## 5. 금지 행동

다음 행위는 허용하지 않는다.

### 5.1 협업 안전 침해

- 인신공격, 조롱, 모욕, 비방, 협박, 따돌림
- 성희롱, 성적 농담, 원치 않는 사적 접근
- 차별적 발언, 혐오 표현, 장애·성별·나이·국적·직군·고용 형태·경제 상황에 대한 비하
- 의도적으로 회의, 리뷰, issue, PR 논의를 방해하는 행위
- 타인의 성과를 무단으로 가로채거나 기여를 지우는 행위
- 반복적인 악의적 리뷰, 근거 없는 비난, 보복성 코멘트

### 5.2 사용자 데이터 침해

- 실제 사용자의 급여, 지출, 저축, 대출, 예산, 납치금액, 계산 스냅샷을 issue, PR, 로그, 문서, 스크린샷, 테스트 데이터에 노출
- 이메일, 전화번호, IP, User-Agent, push token, refresh token, access token, secret key를 원문으로 공유
- 운영 DB 데이터를 로컬, staging, UAT, 데모, 개인 장비, 비인가 저장소로 복사
- 탈퇴·삭제 요청 대상 데이터를 임의로 복구하거나 재사용
- 커뮤니티 익명글 작성자 식별정보를 권한 없이 노출
- 관리자 감사로그를 임의 삭제하거나 조작

### 5.3 보안·권한 침해

- 인증, 인가, RLS, RBAC, rate limit, audit log, idempotency, CSRF/CORS 정책을 우회
- 보안 취약점을 공개 채널에 악용 가능한 형태로 게시
- secret, token, credential, private key를 commit, issue, PR, log, screenshot에 포함
- 승인되지 않은 penetration test, 부하 테스트, 스캐닝, 취약점 재현
- 운영 환경에서 허가 없이 migration, seed, reset, rollback, backup restore 실행
- 보안 경고를 무시하거나 의도적으로 CI/CD 검증을 우회

### 5.4 계산·운영 무결성 침해

- 클라이언트 계산값을 최종 금액으로 확정
- 음수 금액, 소수점 금액, 통화 불일치, 중복 지출 등록을 허용
- 납치금액을 0원 미만으로 표시
- DB transaction, idempotency, snapshot, audit trail 없이 급여·예산·지출 데이터를 변경
- 계산 오류를 알고도 릴리즈하거나 이슈를 축소 보고
- 장애, 배포 실패, 데이터 정합성 문제를 숨김

### 5.5 광고·제휴 정책 위반

- 급여액, 지출액, 저축액, 납치금액 원문을 광고 이벤트나 제휴 타겟팅 데이터에 저장
- 사용자 동의 없이 민감한 재무 데이터를 광고 추천에 활용
- 광고 성과를 위해 사용자 신뢰를 해치는 다크패턴을 설계
- 후원, 광고, 제휴 콘텐츠를 일반 운영 콘텐츠처럼 오인하게 표시
- 광고/제휴 파트너에게 원천 재무 데이터나 식별 가능한 사용자 데이터를 제공

## 6. 민감정보 취급 기준

급여납치 프로젝트에서는 다음 데이터를 민감정보로 본다.

| 등급 | 데이터                                                                        |
| ---- | ----------------------------------------------------------------------------- |
| S1   | 급여액, 고정지출, 고정저축, 일일예산, 변동지출, 납치금액, 계산 스냅샷         |
| S2   | 이메일, 전화번호, 소셜 로그인 식별자, 기기 식별자, push token hash, 동의 이력 |
| S3   | 알림, 커뮤니티 게시글/댓글/신고, 첨부 파일, 마이페이지 정보                   |
| S4   | 관리자 감사로그, 운영 장애 기록, CS 기록, 보안 신고                           |
| S5   | 광고 이벤트, 캠페인, 파트너, 비식별 성과 집계                                 |

민감정보 취급 원칙은 다음과 같다.

1. 필요한 목적과 권한이 없으면 접근하지 않는다.
2. 원문 대신 hash, mask, pseudonym, aggregate 값을 우선 사용한다.
3. 로컬·staging·UAT 환경에는 실제 개인정보와 실제 금융정보를 사용하지 않는다.
4. 로그에는 원문 민감정보를 남기지 않는다.
5. 광고/제휴 데이터는 재무 원천 데이터와 분리한다.
6. 보안 신고와 incident 기록은 비공개 채널에서 관리한다.
7. 삭제·보존 정책은 `database/policies/retention-policy.md` 기준을 따른다.

## 7. 커뮤니티 안전 기준

급여납치 커뮤니티는 직장인, 아르바이트생, 사회초년생, 소비 통제와 자기계발을 원하는 사용자가 정보를 공유하는 공간이다.

커뮤니티 기능을 개발·운영할 때 다음 기준을 지킨다.

1. 급여, 부업, 투자, 대출, 소비, 건강, 자기계발 관련 정보는 오해를 줄 수 있으므로 과장·허위·불법 조언을 방치하지 않는다.
2. 투자 수익 보장, 대출 권유, 불법 금융, 도박, 사행성, 불법 부업, 피싱, 리워드 악용 콘텐츠를 금지한다.
3. 익명 기능은 사용자의 화면 표시 익명성을 보호하되, 신고·법적 대응·운영 감사에 필요한 내부 소유권은 안전하게 유지한다.
4. 신고, 숨김, 제재, 복구, 이의제기 절차를 문서화한다.
5. 운영자와 관리자는 개인적 취향이 아니라 문서화된 정책 기준으로 조치한다.
6. 커뮤니티 운영 데이터와 사용자의 재무 원천 데이터를 결합하지 않는다.

## 8. 보안 취약점 신고

보안 취약점은 공개 issue, 공개 PR, 공개 discussion, 커뮤니티 채널에 게시하지 않는다.

신고 대상 예시는 다음과 같다.

- 인증 우회
- 권한 상승
- RLS 우회
- 관리자 RBAC 우회
- 급여·예산·지출 계산 오류
- 민감정보 노출
- secret 또는 token 노출
- 광고 이벤트와 재무 원천 데이터 결합 위험
- 커뮤니티 익명성 침해
- push token 또는 알림 payload 노출
- SQL injection, XSS, SSRF, CSRF, RCE, dependency vulnerability
- backup, restore, log, analytics, monitoring 경로의 데이터 유출 위험

신고자는 `SECURITY.md`에 정의된 비공개 보안 신고 절차를 따른다.

보안 신고에는 가능한 범위에서 다음 정보를 포함한다.

- 영향받는 영역
- 재현 조건
- 예상 영향
- 관련 requestId 또는 traceId
- 노출 가능 데이터 유형
- 권한 수준
- 스크린샷 또는 로그가 필요한 경우 마스킹 처리된 자료
- 악용 가능성 평가
- 임시 완화 방안

---

## 9. 리뷰와 의사결정 기준

PR, issue, 문서 변경, 설계 변경을 검토할 때 다음 기준을 적용한다.

1. 요구사항과 사용자 가치가 명확한가?
2. 급여·예산·지출·저축 계산이 서버 권위인가?
3. DB migration, seed, rollback, backup 기준과 충돌하지 않는가?
4. RLS, RBAC, audit log, idempotency가 유지되는가?
5. 민감정보가 로그, 문서, 테스트, 광고 이벤트에 노출되지 않는가?
6. 모바일 앱, 관리자 콘솔, API, scheduler, notification, DB, 운영 문서 간 계약이 일치하는가?
7. 접근성, 반응형 UI, 오류 상태, 빈 상태, 로딩 상태가 고려되었는가?
8. 성능 목표와 관측성 기준이 반영되었는가?
9. QA, E2E, 보안, 개인정보 검증이 가능한가?
10. 장애 발생 시 rollback 또는 완화 방안이 있는가?

---

## 10. 위반 신고

행동강령 위반은 공개 망신주기나 보복을 피하고 정해진 비공개 절차로 신고한다.

신고 내용에는 가능한 범위에서 다음 정보를 포함한다.

- 발생 일시
- 발생 위치
- 관련 issue, PR, commit, 회의, 채팅, 문서
- 위반 유형
- 영향받은 사람 또는 데이터
- 캡처, 로그, 링크 등 증거 자료
- 즉시 필요한 보호 조치
- 신고자가 원하는 후속 연락 방식

신고자와 피해자 보호가 우선이며, 보복 행위는 별도의 중대한 위반으로 처리한다.

---

## 11. 집행 절차

프로젝트 유지관리자와 운영 책임자는 위반 신고를 받으면 다음 순서로 처리한다.

1. 신고 접수 확인
2. 즉시 위험 여부 판단
3. 피해자와 신고자 보호 조치
4. 관련 자료 보존
5. 이해상충이 있는 담당자 배제
6. 사실 확인
7. 위반 수준 분류
8. 조치 결정
9. 당사자에게 필요한 범위의 결과 안내
10. 재발 방지 대책 수립

보안, 개인정보, 재무 데이터, 운영 DB, 광고 데이터 분리 위반은 일반 행동 위반보다 높은 위험도로 판단한다.

---

## 12. 조치 수준

위반 수준에 따라 다음 조치를 적용할 수 있다.

| 수준    | 예시                                                     | 조치                                                  |
| ------- | -------------------------------------------------------- | ----------------------------------------------------- |
| Level 1 | 부주의한 표현, 경미한 협업 태도 문제                     | 비공개 주의, 수정 요청, 가이드 제공                   |
| Level 2 | 반복적 무례, 근거 없는 공격, 경미한 정책 위반            | 공식 경고, 리뷰 제한, 일정 기간 참여 제한             |
| Level 3 | 민감정보 부주의 노출, 보안 절차 미준수, 차별적 발언      | 접근권한 제한, PR/issue 참여 제한, 보안 재교육        |
| Level 4 | 고의적 데이터 노출, 권한 우회, 괴롭힘, 보복, secret 유출 | 즉시 접근 차단, incident 처리, 관리자 감사 기록       |
| Level 5 | 악의적 침해, 데이터 탈취, 불법 행위, 운영 환경 공격      | 영구 차단, 법적 대응, 관계 기관 또는 계약상 절차 진행 |

조치 수준은 위반의 의도, 반복성, 영향 범위, 데이터 민감도, 사용자 피해 가능성, 복구 가능성, 협조 여부를 기준으로 결정한다.

## 13. 유지관리자와 관리자의 책임

프로젝트 유지관리자, 관리자, 운영자는 더 높은 책임을 가진다.

관리 책임자는 다음을 지켜야 한다.

1. 권한을 개인적 갈등 해결에 사용하지 않는다.
2. 신고자를 보호하고 보복을 방지한다.
3. 보안·개인정보 이슈를 축소하거나 은폐하지 않는다.
4. 관리자 기능 사용 내역은 감사 가능해야 한다.
5. 운영자 판단은 문서화된 정책과 근거를 따른다.
6. 제재와 복구는 일관성 있게 수행한다.
7. 실수로 인한 위반도 영향이 크면 incident로 처리한다.
8. 권한 부여와 회수는 최소 권한 원칙을 따른다.
