# 급여납치 Mobile Salary Feature README

문서 버전: v3.1.0  
대상 경로: `apps/mobile/src/features/salary/README.md`  
모듈 범위: 급여 현황, 납치금액, 누적 납치금액, 급여 홈 요약, 서버 권위 급여 계산, 예산·지출·저축 연동, 알림·LV UP·커뮤니티 연결, 광고·제휴 개인정보 보호 경계

## 1. 모듈 목적

Salary Feature는 급여납치 모바일 앱에서 사용자가 가장 먼저 확인하는 급여 중심 대시보드 모듈이다. 이 모듈은 사용자의 급여를 단순히 표시하는 화면이 아니라, 서버에서 확정한 급여 계획과 실제 지출·저축·예산 데이터를 기반으로 현재 급여 주기에서 얼마나 안전하게 돈을 지키고 있는지 보여주는 핵심 운영 영역이다.

본 모듈은 다음을 담당한다.

1. 급여 현황 표시
2. 예상 급여 납치금액 표시
3. 실제 누적 납치금액 표시
4. 오늘 남은 생활비 표시
5. 고정지출·고정저축·변동지출 요약
6. 급여 계획 재계산 요청
7. 일일 예산 초과 경고 연결
8. 급여일·월마감 알림 연결
9. LV UP 루틴과 절약 행동 연결
10. 커뮤니티 인증글 이동 연결
11. 광고·제휴 영역의 금융 타겟팅 차단
12. 개인정보·금융 원시 데이터 노출 방지

## 2. 핵심 원칙

### 2.1 서버 권위 원칙

급여, 예산, 지출, 저축, 납치금액 계산은 클라이언트가 최종 판단하지 않는다. 모바일 앱은 서버에서 계산된 결과를 조회하고 표시하며, 사용자의 요청은 서버 API를 통해 검증·계산·저장된다.

클라이언트에서 허용되는 계산은 다음 범위로 제한한다.

- 화면 표시용 포맷팅
- 정수 KRW의 안전한 마스킹
- 서버 응답의 음수 방어 표시
- 오프라인 fallback용 비권위 placeholder
- skeleton, empty state, loading state 처리

클라이언트에서 금지되는 계산은 다음과 같다.

- 실제 급여 납치금액 최종 계산
- 월마감 실제 납치금액 확정
- 고정지출·고정저축 차감 확정
- 예산 초과 상태의 최종 판정
- 광고 타겟팅용 금융 세그먼트 산출

### 2.2 KRW 정수 원칙

모든 금액은 원화 정수 단위로 처리한다.

- 소수점 금지
- 음수 금지
- `NaN`, `Infinity` 금지
- 클라이언트 표시 전 `Math.trunc` 또는 서버 정규화 결과 사용
- 원시 금액 로그 금지
- 광고·제휴 payload로 원시 금액 전달 금지

### 2.3 개인정보 보호 원칙

Salary Feature는 민감도가 가장 높은 급여 데이터를 다루므로 다음 값을 화면 외부로 유출하지 않는다.

- 급여 원시 금액
- 납치금액 원시 금액
- 지출·저축 원시 상세
- 계좌·카드·이메일·전화번호
- 인증 토큰·세션·푸시 토큰
- 사용자 식별자 원문

로그, 오류 메시지, 광고 이벤트, 분석 이벤트, 커뮤니티 이동 payload에는 위 정보가 포함되면 안 된다.

## 3. 서버 공식 계산식

Salary Feature가 참조하는 공식 계산식은 서버 API가 권위 있게 수행한다.

```text
planned_total_expense = fixed_expense + daily_living_budget + planned_other_expense
expected_hijack = max(0, expected_salary - planned_total_expense)
today_remaining_budget = max(0, daily_limit - today_variable_expense)
monthly_actual_hijack = max(0, actual_income + carry_over - actual_expense - actual_savings)
```

모바일 앱은 위 결과를 서버 응답으로 받는다. 클라이언트는 표시상 방어를 위해 `max(0, value)` 형태의 안전 보정은 할 수 있으나, 이를 DB 저장값이나 최종 산출값으로 간주하지 않는다.

## 4. 주요 화면 책임

### 4.1 급여 홈 Hero

급여 홈 Hero는 현재 급여 주기의 핵심 상태를 표시한다.

- 이번 달 예상 급여
- 예상 납치금액
- 실제 누적 납치금액
- 오늘 남은 생활비
- 급여일 D-day
- 월마감 상태
- 데이터 동기화 상태

Hero 영역에는 사용자의 원시 금융 데이터가 외부 SDK나 광고 SDK에 전달되지 않아야 한다.

### 4.2 급여 계획 요약

급여 계획 요약은 다음 항목을 표시한다.

- 급여 주기 시작일과 종료일
- 예상 급여
- 고정지출 합계
- 고정저축 합계
- 생활비 계획
- 기타 계획 지출
- 예상 납치금액
- 서버 재계산 시각

계획 수정은 Salary Feature가 직접 처리하지 않고 Plan Feature 또는 서버 계획 API로 이동한다.

### 4.3 오늘 예산 상태

오늘 예산 상태는 Budget Feature와 연동된다.

- 일일 한도
- 오늘 사용액
- 오늘 남은 금액
- 초과 여부
- 초과 시 경고 메시지
- 변동지출 추가 버튼

최종 초과 판정은 서버 API가 제공한 값을 우선한다.

### 4.4 고정지출·고정저축 요약

Salary Feature는 전체 등록·수정 화면이 아니라 요약과 이동을 담당한다.

- 이번 주기 고정지출 예정 합계
- 납부 완료 수
- 미납 수
- 고정저축 예정 합계
- 실행 완료 수
- 실패 또는 보류 수

세부 CRUD는 Plan, Expense, Savings Feature로 위임한다.

### 4.5 납치금액 카드

납치금액 카드는 급여납치 서비스의 핵심 가치 지표다.

- 예상 납치금액
- 실제 누적 납치금액
- 전월 대비 변화
- 월마감 확정 여부
- 목표 대비 달성률

마케팅·광고·커뮤니티 공유에서는 원시 금액을 그대로 내보내지 않고, 사용자가 명시적으로 선택한 범위형·마스킹형 표현만 허용한다.

## 5. API 계약

### 5.1 조회 API

```http
GET /api/v1/payroll/home
```

권장 응답 구조는 다음과 같다.

```json
{
  "data": {
    "cycle": {
      "id": "cycle_id",
      "startDate": "2026-06-01",
      "endDate": "2026-06-30",
      "payday": "2026-06-25",
      "status": "OPEN"
    },
    "summary": {
      "expectedSalary": 3000000,
      "fixedExpenseTotal": 1200000,
      "fixedSavingsTotal": 500000,
      "dailyLivingBudgetTotal": 600000,
      "plannedOtherExpense": 100000,
      "expectedHijack": 600000,
      "actualHijack": 350000,
      "todayRemainingBudget": 18000
    },
    "alerts": [],
    "contextualAd": null,
    "privacy": {
      "rawFinancialDataExposed": false,
      "rawPersonalDataExposed": false,
      "adsFinancialTargetingUsed": false
    }
  }
}
```

### 5.2 재계산 API

```http
POST /api/v1/payroll/plans/:planId/recalculate
```

재계산 요청은 서버에서 다음을 수행해야 한다.

- 사용자 소유권 확인
- 활성 급여 주기 확인
- 고정지출 합계 재산출
- 고정저축 합계 재산출
- 변동지출 합계 재산출
- 일일 예산 잔액 재산출
- 예상 납치금액 재산출
- 감사 로그 기록
- 알림 이벤트 생성 여부 판단

### 5.3 월마감 API

월마감 확정은 모바일 화면에서 임의 계산하지 않는다. 서버 스케줄러 또는 명시적 서버 API가 담당한다.

```http
POST /api/v1/payroll/cycles/:cycleId/close
```

필수 조건:

- 인증된 사용자 또는 시스템 권한
- 중복 마감 방지 idempotency key
- 실제 수입·지출·저축 정산
- 월마감 감사 로그
- 확정 후 읽기 전용 상태 전환

## 6. 상태 모델

### 6.1 Loading

- 최초 진입
- 서버 재조회
- pull-to-refresh
- 재계산 요청 중

### 6.2 Ready

- 정상 급여 데이터 표시
- 서버 계산 결과 표시
- 액션 버튼 활성화

### 6.3 Empty

- 급여 계획 미등록
- 온보딩 미완료
- 첫 급여 주기 미생성

이 경우 Plan Feature 또는 Onboarding으로 이동한다.

### 6.4 Error

- 네트워크 오류
- 인증 만료
- 서버 검증 실패
- 권한 없음
- 점검 모드

오류 메시지는 민감 데이터를 제거한 안전 문구만 표시한다.

### 6.5 Offline

- 마지막 안전 snapshot 표시 가능
- 원시 금융 데이터 저장 금지
- 서버 권위 값이 아님을 명확히 표시
- 쓰기 요청 차단 또는 queue 처리 금지

## 8. 광고·제휴 정책

Salary Feature에서 광고·제휴를 노출할 수 있으나 다음 원칙을 반드시 지킨다.

1. 급여 금액 기반 타겟팅 금지
2. 납치금액 기반 타겟팅 금지
3. 지출·저축 상세 기반 타겟팅 금지
4. 사용자 식별자 원문 전달 금지
5. 광고주에게 raw financial data 제공 금지
6. 광고·제휴 라벨 명확 표시
7. 기본값은 contextual 또는 non-personalized
8. 광고 클릭 이벤트에도 금융 금액 제외
9. 광고 추천 사유에 민감 정보 노출 금지
10. 광고 SDK 로그에 민감 정보 포함 금지

## 10. 보안 요구사항

Salary Feature는 다음 보안 요구사항을 만족해야 한다.

- 모든 쓰기 요청은 인증 필요

- 클라이언트 계산 결과 저장 금지
- 재계산 요청 idempotency 고려
- 장기 캐시에 raw 금융 데이터 저장 금지
- 스크린샷·공유 기능 사용 시 민감정보 경고

## 13. 완료 기준

이 README 기준으로 Salary Feature가 완료되려면 다음 조건을 만족해야 한다.

1. 급여 홈 화면이 존재한다.
2. 서버 권위 API를 사용한다.
3. 급여·예산·지출·저축·납치금액 핵심 지표가 표시된다.
4. 재계산 요청이 서버 API로 전달된다.
5. KRW 정수 원칙이 지켜진다.
6. 음수 금액이 표시되지 않는다.
7. 민감 정보가 로그·광고·커뮤니티 payload로 유출되지 않는다.
8. 광고 금융 타겟팅이 금지된다.
9. 커뮤니티 공유 시 원시 금액이 제거된다.
10. 인증·소유권 검증을 전제로 한다.
11. 알림·예산·계획·지출·저축·LV UP·프로필과 연결된다.
12. 로딩·빈 상태·오류·오프라인 상태가 정의된다.
