# 03. 소셜 로그인 연동 문서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼에서 네이버, 카카오, 구글, 애플, 페이스북 소셜 로그인을 구현하기 위한 OAuth/OIDC 연동 기준을 최종 확정한다. 모바일 앱은 provider 인증을 수행하고, 백엔드는 authorization code 또는 id token을 검증해 내부 사용자 세션을 발급한다.

## 2. 공통 OAuth 원칙

| 항목           | 최종 기준                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| 방식           | Authorization Code Flow + PKCE를 기본으로 한다.                                                               |
| CSRF 방지      | `state`는 서버 또는 앱 보안 저장소에 저장 후 callback 시 반드시 비교한다.                                     |
| PKCE           | `code_challenge=S256`, `code_verifier` 필수. provider 미지원 시 state와 redirect allowlist로 보완한다.        |
| Redirect URI   | 모바일 딥링크와 백엔드 콜백 URI를 provider 콘솔에 모두 등록한다.                                              |
| 계정 식별      | 이메일이 아닌 provider별 고유 ID를 기본 식별자로 사용한다.                                                    |
| 이메일 처리    | provider가 이메일을 제공하지 않거나 변경할 수 있으므로 보조 정보로만 사용한다.                                |
| 내부 계정 연결 | `auth_identities(provider, provider_user_key)` unique 기준으로 users와 연결한다.                              |
| 토큰 저장      | provider access token은 기본 저장하지 않는다. 장기 API 호출이 필요한 경우 별도 동의와 암호화 저장이 필요하다. |

| GOOGLE   | `https://accounts.google.com/o/oauth2/v2/auth`     | `https://oauth2.googleapis.com/token`                      | `https://openidconnect.googleapis.com/v1/userinfo` | GOOGLE           |

## 4. 모바일 앱 로그인 플로우

1. 사용자가 로그인 화면에서 provider 버튼을 누른다.
2. 앱은 `state`, `code_verifier`, `code_challenge`를 생성한다.
3. provider 인증 URL로 이동한다.
4. 사용자가 provider에서 동의한다.
5. provider는 등록된 redirect URI로 `code`, `state`를 반환한다.
6. 앱은 state를 검증한다.
7. 앱은 백엔드 `POST /auth/social-login`으로 provider, code, redirectUri, codeVerifier, deviceId를 전달한다.
8. 백엔드는 provider 토큰 URL로 code를 교환한다.
9. 백엔드는 사용자 정보 또는 ID Token을 검증한다.
10. 내부 사용자와 `auth_identities`를 조회한다.
11. 기존 연결이 있으면 로그인, 없으면 신규 users/auth_identities/user_profiles/user_settings를 생성한다.
12. 내부 access token, refresh token을 발급한다.

## 5. 백엔드 API

### 5.1 소셜 로그인 요청

`POST /api/v1/auth/social-login`

```json
{
  "provider": "NAVER",
  "authorizationCode": "provider_code",
  "redirectUri": "salaryhijacking://oauth/naver",
  "codeVerifier": "plain-pkce-code-verifier",
  "state": "state-from-provider",
  "deviceId": "device_ios_001",
  "autoLogin": true
}
```

### 5.2 응답

```json
{
  "success": true,
  "data": {
    "userId": "uuid",
    "isNewUser": false,
    "provider": "NAVER",
    "onboardingCompleted": true,
    "accessToken": "jwt.access.token",
    "refreshToken": "jwt.refresh.token",
    "expiresIn": 900,
    "nextScreen": "HOME"
  }
}
```

## 7. 계정 연결 정책

| 상황                                      | 처리                                                                                          |
| ----------------------------------------- | --------------------------------------------------------------------------------------------- |
| 동일 provider_user_key 존재               | 해당 users로 로그인                                                                           |
| provider_user_key 없음                    | `AUTH_PROVIDER_ID_MISSING` 오류                                                               |
| provider 이메일이 기존 이메일 계정과 동일 | 자동 병합하지 않는다. 사용자 확인 후 계정 연결 API로 처리한다.                                |
| 동일 사용자가 여러 provider 연결          | `auth_identities`에 provider별 row 추가                                                       |
| provider 연동 해제                        | 내부 refresh token 폐기, auth_identity.revoked_at 기록, provider revoke API 호출 가능 시 호출 |

## 8. 보안 검증

1. redirectUri는 allowlist와 정확히 일치해야 한다.
2. state는 요청 시 생성한 값과 일치해야 한다.
3. PKCE code_verifier는 최초 code_challenge와 일치해야 한다.
4. Google/Apple ID Token은 issuer, audience, expiration, nonce를 검증한다.
5. provider access token으로 사용자 정보를 조회할 때 HTTPS만 허용한다.
6. provider_user_key는 절대 클라이언트 입력값만 신뢰하지 않는다.
7. provider 이메일만으로 기존 계정 자동 병합 금지.

## 12. 외부 기준

Provider OAuth endpoint와 흐름은 각 provider 공식 문서 기준으로 확인하고, 본 문서는 2026-06-15 기준 구현 계약으로 고정한다. 운영 중 provider 정책이 변경될 경우 코드 수정은 가능하지만 내부 API 계약은 본 문서 기준을 유지한다.

## 13. 완료 판정

본 문서는 provider별 OAuth flow, callback, 사용자 정보 매핑, 보안 검증, 오류 처리, 테스트 조건을 포함하므로 소셜 로그인 구현의 최종본으로 확정한다.
