# 02. 인증/인가 명세서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 이메일 로그인, SNS 로그인, 토큰, 세션, 권한, 계정 상태, 보안 정책을 최종 확정한다. 급여·지출·저축 정보는 민감도가 높은 개인 재무 데이터이므로 인증/인가 정책은 모든 사용자 API의 전제 조건이다.

## 2. 인증 방식

| 방식            | 적용 범위                            | 최종 기준                                              |
| --------------- | ------------------------------------ | ------------------------------------------------------ |
| 이메일/비밀번호 | 일반 회원가입/로그인                 | 비밀번호 해시 저장, 로그인 실패 제한, 이메일 중복 방지 |
| 소셜 OAuth      | 네이버, 카카오, 구글, 애플, 페이스북 | provider 식별자 기반 계정 연결, PKCE/State 검증        |
| Access Token    | 사용자 API 호출                      | JWT, 짧은 만료시간, Bearer 방식                        |
| Refresh Token   | 자동 로그인/세션 유지                | DB 저장 해시, rotation, 기기 단위 폐기                 |
| Admin Token     | 관리자 API                           | 관리자 역할과 2단계 인증 적용                          |
| Service Token   | 배치/내부 작업                       | 내부 네트워크/서명 검증, 사용자 API 접근 불가          |

## 3. 계정 상태 모델

| 상태      | 의미           | 로그인 | API 접근 | 전환 조건              |
| --------- | -------------- | -----: | -------: | ---------------------- |
| ACTIVE    | 정상 사용자    |   가능 |     가능 | 가입/정지 해제         |
| SUSPENDED | 운영자 제재    |   불가 |     불가 | 신고/정책 위반 처리    |
| WITHDRAWN | 탈퇴 완료      |   불가 |     불가 | 사용자의 탈퇴 확정     |
| DELETED   | 법적/운영 삭제 |   불가 |     불가 | 보존 기간 종료 후 삭제 |

## 4. 권한 모델 RBAC

| Role        | 설명          | 허용 권한                                     |
| ----------- | ------------- | --------------------------------------------- |
| GUEST       | 비회원        | 로그인, 회원가입, 공개 앱 정보                |
| USER        | 일반 사용자   | 본인 급여/지출/커뮤니티/레벨업 데이터 접근    |
| OPERATOR    | 운영자        | 신고, 게시글 숨김, 공지/배너 조회·수정 일부   |
| ADMIN       | 관리자        | 운영자 권한 + 사용자 제재, 배너/공지 관리     |
| SUPER_ADMIN | 최고관리자    | 전체 관리자 권한, 권한 부여/회수, 시스템 설정 |
| SYSTEM      | 배치/스케줄러 | 알림 생성, 월말 확정, 통계 집계               |

## 5. 토큰 정책

| 항목       | Access Token                             | Refresh Token                                 |
| ---------- | ---------------------------------------- | --------------------------------------------- |
| 형식       | JWT                                      | 난수 토큰 또는 JWT, DB에는 해시 저장          |
| 만료       | 15분                                     | 30일, 자동 로그인 선택 시 90일                |
| 저장 위치  | 앱 보안 저장소                           | 앱 보안 저장소                                |
| 재발급     | 불가                                     | `/auth/refresh`로 rotation                    |
| 폐기       | 만료 또는 로그아웃                       | 로그아웃, 탈퇴, 비밀번호 변경, 의심 행위 감지 |
| 포함 claim | sub, role, sessionId, deviceId, iat, exp | sessionId, deviceId, token family             |

JWT Claim 예시

```json
{
  "iss": "salary-hijacking-api",
  "aud": "salary-hijacking-mobile",
  "sub": "0c2f2f1a-0e6e-4cd7-a2dc-03ef3e0c1111",
  "role": "USER",
  "sessionId": "sess_01JZ...",
  "deviceId": "device_ios_001",
  "iat": 1781492400,
  "exp": 1781493300
}
```

## 6. 세션 테이블 기준

| 필드                 | 설명                                                        |
| -------------------- | ----------------------------------------------------------- |
| session_id           | 사용자 로그인 세션 ID                                       |
| user_id              | 사용자 ID                                                   |
| device_id            | 앱 설치/기기 식별자                                         |
| refresh_token_hash   | refresh token 해시                                          |
| refresh_token_family | rotation 계열 식별자                                        |
| ip_address           | 마지막 요청 IP                                              |
| user_agent           | 앱/OS 정보                                                  |
| expires_at           | refresh token 만료시각                                      |
| revoked_at           | 폐기 시각                                                   |
| revoked_reason       | LOGOUT, WITHDRAW, PASSWORD_CHANGED, ROTATION_REUSE_DETECTED |

## 7. 이메일 회원가입 정책

| 항목      | 기준                                                                                  |
| --------- | ------------------------------------------------------------------------------------- |
| 이메일    | RFC 형식, 활성 사용자 중 unique, 대소문자 구분 없이 정규화                            |
| 비밀번호  | 10자 이상, 영문/숫자/특수문자 중 2종 이상, 이메일과 동일 문자열 금지                  |
| 닉네임    | 2~20자, 금칙어/관리자 사칭 금지, 중복 가능 여부는 정책상 허용하되 내부 user_id로 식별 |
| 필수 동의 | 이용약관, 개인정보처리방침                                                            |
| 선택 동의 | 마케팅 수신, 광고/제휴 혜택 수신                                                      |

## 8. 로그인 정상 흐름

1. 사용자가 이메일/비밀번호 또는 소셜 로그인을 요청한다.
2. 서버는 계정 상태, 비밀번호/소셜 토큰, rate limit을 검증한다.
3. 인증 성공 시 access token, refresh token, sessionId를 발급한다.
4. `last_login_at`을 갱신한다.
5. 사용자의 급여 계획 존재 여부를 확인한다.
6. `onboardingCompleted`가 false이면 초기 급여 설정 화면으로 이동하게 한다.
7. true이면 급여 홈으로 이동하게 한다.

## 9. 로그인 실패 정책

| 실패 상황            | 처리                                         |
| -------------------- | -------------------------------------------- |
| 비밀번호 불일치      | `AUTH_INVALID_CREDENTIALS`, 상세 원인 비노출 |
| 존재하지 않는 이메일 | 동일하게 `AUTH_INVALID_CREDENTIALS` 반환     |
| 5회 이상 실패        | IP+이메일 기준 15분 로그인 제한              |
| 정지 계정            | `USER_SUSPENDED`, 고객센터 안내              |
| 탈퇴 계정            | `USER_WITHDRAWN`, 재가입 정책 안내           |
| 소셜 토큰 만료       | `AUTH_SOCIAL_TOKEN_EXPIRED`, 재인증 유도     |
| provider 식별자 없음 | `AUTH_PROVIDER_ID_MISSING`, 가입 실패 처리   |

## 10. 인가 규칙

| 리소스          | 사용자 접근                                    | 관리자 접근                                                     |
| --------------- | ---------------------------------------------- | --------------------------------------------------------------- |
| 급여/지출/저축  | 본인 `user_id` 일치 시만 가능                  | 원칙적으로 직접 열람 금지, 고객 문의/분쟁 처리 시 감사로그 필수 |
| 커뮤니티 게시글 | 공개 게시글 조회 가능, 작성자만 수정/삭제 가능 | 숨김/복구/삭제/제재 가능                                        |
| 알림            | 본인 알림만 가능                               | 공지성 알림 발송 가능                                           |
| 광고 로그       | 본인 로그 직접 조회 불가                       | 통계성 집계만 가능                                              |
| 관리자 설정     | 접근 불가                                      | ADMIN 이상                                                      |

## 11. 민감 데이터 보호 기준

1. 비밀번호는 Argon2id 또는 bcrypt로 해시하며 평문 저장 금지.
2. refresh token은 원문 저장 금지, 해시만 저장.
3. 급여/대출/저축/지출 데이터는 광고 타겟팅 데이터와 분리.
4. 관리자 조회는 최소 권한, 사유 입력, 감사로그 기록 필수.
5. 탈퇴 사용자의 인증 토큰과 세션은 즉시 폐기.

## 12. API 인증 오류 코드

| 코드                   | HTTP | 메시지                             | 처리             |
| ---------------------- | ---: | ---------------------------------- | ---------------- |
| AUTH_TOKEN_MISSING     |  401 | 인증 정보가 없습니다.              | 로그인 화면 이동 |
| AUTH_TOKEN_EXPIRED     |  401 | 인증 시간이 만료되었습니다.        | refresh 시도     |
| AUTH_TOKEN_INVALID     |  401 | 유효하지 않은 인증 정보입니다.     | 재로그인         |
| AUTH_REFRESH_EXPIRED   |  401 | 자동 로그인 기간이 만료되었습니다. | 재로그인         |
| AUTH_REFRESH_REUSED    |  401 | 보안상 세션이 만료되었습니다.      | 전체 세션 폐기   |
| AUTH_PERMISSION_DENIED |  403 | 접근 권한이 없습니다.              | 접근 차단        |
| USER_SUSPENDED         |  403 | 이용이 제한된 계정입니다.          | 고객센터 안내    |
