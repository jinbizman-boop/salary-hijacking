# 08. 배치/스케줄러 설계서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 자동 알림, 월별 계산, 미사용 파일 정리, 토큰 정리, 통계 집계를 수행하는 배치/스케줄러 구조를 최종 확정한다. 배치 작업은 사용자 경험의 일관성과 금액 계산 정확성을 보장하기 위한 백엔드 핵심 구성요소다.

## 2. 스케줄러 실행 환경

| 항목      | 최종 기준                                                         |
| --------- | ----------------------------------------------------------------- |
| 실행 방식 | Cloudflare Workers Cron Triggers 또는 백엔드 작업자 프로세스      |
| DB        | Neon DB / PostgreSQL                                              |
| 타임존    | 사용자 기준 알림 판단은 Asia/Seoul, 저장은 UTC                    |
| 중복 방지 | 작업별 `job_key` unique, advisory lock 또는 job_locks 테이블 사용 |
| 재시도    | 실패 시 exponential backoff, 최대 3회, 이후 FAILED                |
| 감사      | 모든 배치 실행은 `batch_job_runs`에 기록                          |
| 멱등성    | 동일 사용자·동일 날짜·동일 이벤트는 중복 생성 금지                |

## 4. JOB-001 급여일 알림 생성

### 목적

사용자의 급여일 전후 행동을 유도한다.

### 대상 조건

1. `payroll_plans.status = ACTIVE`
2. `payday`가 오늘 또는 내일인 사용자
3. `user_settings.push_enabled = true`
4. 동일 `dedupe_key` 알림이 존재하지 않음

### 생성 알림

| 조건           | Type            | 메시지                                                 |
| -------------- | --------------- | ------------------------------------------------------ |
| 급여일 하루 전 | PAYDAY_TOMORROW | 내일 급여일이에요. 월급 납치 계획을 확인해보세요.      |
| 급여일 당일    | PAYDAY_TODAY    | 오늘 급여일이에요. 수령금액과 지출계획을 확인해보세요. |

## 5. JOB-002 고정지출 예정 알림

### 로직

1. 오늘 날짜의 일(day)을 구한다.
2. `fixed_expenses.expense_day`가 오늘 또는 내일과 일치하는 항목을 조회한다.
3. 상태가 `SCHEDULED`인 항목만 대상이다.
4. 사용자 설정 `fixed_payment_alert_enabled`가 true인지 확인한다.
5. 알림 `FIXED_PAYMENT_DUE`를 생성한다.
6. `dedupe_key = fixed_due:{userId}:{fixedExpenseId}:{date}`를 적용한다.

## 6. JOB-003 고정저축 예정 알림

고정지출과 동일한 구조로 `savings_plans.saving_day`, 상태 `SCHEDULED`, 알림 `SAVINGS_TRANSFER_DUE`를 기준으로 생성한다.

## 7. JOB-004 일일 예산 상태 점검

### 계산

```text
remaining_amount = daily_limit_amount - used_amount
remaining_rate = remaining_amount / daily_limit_amount * 100
```

### 알림 조건

| 조건                                           | 알림                 |
| ---------------------------------------------- | -------------------- |
| remaining_amount < 0                           | BUDGET_OVER          |
| remaining_rate <= 20 and remaining_amount >= 0 | BUDGET_REMAINING_LOW |
| daily_limit_amount = 0                         | 알림 생성 안 함      |

동일 날짜, 동일 사용자, 동일 type은 1회만 생성한다.

## 8. JOB-005 레벨업 미션 알림

| 시간  | 메시지                           |
| ----- | -------------------------------- |
| 09:00 | 오늘의 레벨업 미션이 도착했어요. |

정책: 사용자가 당일 이미 모든 미션을 완료했다면 발송하지 않는다. `growth_alert_enabled=false`이면 앱 내 알림과 푸시 모두 생성하지 않는다.

## 9. JOB-006 월말 납치금액 확정

### 확정 대상

1. 전월 `payroll_plans.status = ACTIVE`
2. 아직 `CLOSED`가 아닌 계획
3. 월 종료 후 1일 00:30 KST 실행
4. 사용자별 수동 마감 API도 동일 로직 사용

### 계산식

```text
actual_fixed_expense_total = sum(fixed_expenses.amount where status in PAID,SCHEDULED and not CANCELLED)
actual_savings_total = sum(savings_plans.amount where status in TRANSFERRED,SCHEDULED and not CANCELLED)
actual_variable_expense_total = sum(variable_expenses.amount where spent_at in month and deleted_at is null)
actual_expense_amount = actual_fixed_expense_total + actual_savings_total + actual_variable_expense_total
confirmed_hijack_amount = max(0, expected_salary_amount - actual_expense_amount)
```

### 처리

1. `payroll_plans.confirmed_hijack_amount` 업데이트
2. `payroll_plans.status = CLOSED`
3. 사용자 누적 납치금액 갱신
4. 목표 달성 알림 조건 확인
5. 감사/계산 로그 기록

## 10. JOB-007 누적 납치금액 재집계

목적: 이벤트 처리 실패 또는 수동 수정으로 인한 집계 불일치를 보정한다.

```sql
SELECT user_id, COALESCE(SUM(confirmed_hijack_amount), 0)
FROM payroll_plans
WHERE status = 'CLOSED'
GROUP BY user_id;
```

결과를 사용자 성과 집계 테이블에 upsert한다.

## 11. JOB-008 미사용 파일 정리

| 대상         | 조건                                                         | 처리                           |
| ------------ | ------------------------------------------------------------ | ------------------------------ |
| PENDING      | created_at < now() - interval '24 hours'                     | EXPIRED 후 R2 object 삭제 시도 |
| READY 미연결 | created_at < now() - interval '7 days' and linked_at is null | DELETED 처리                   |
| DELETED      | object_delete_status != DONE                                 | R2 삭제 재시도                 |

## 12. JOB-009 만료 세션 정리

1. `expires_at < now()`인 session 조회.
2. `revoked_at is null`이면 `revoked_at=now()`, reason=`EXPIRED` 처리.
3. device token은 로그아웃이 아닌 만료만으로 삭제하지 않는다.

## 14. JOB-011 신고 자동 숨김 점검

정책: 신고 누적 임계치가 도달하면 관리자 검토 전까지 임시 숨김 처리한다.

| 조건                           | 처리                                |
| ------------------------------ | ----------------------------------- |
| 동일 게시글 유효 신고 5건 이상 | status = HIDDEN_PENDING_REVIEW      |
| 개인정보 노출 신고 1건 이상    | 우선 숨김 후 검토                   |
| 명백한 스팸 패턴               | 자동 숨김 및 작성자 rate limit 강화 |
