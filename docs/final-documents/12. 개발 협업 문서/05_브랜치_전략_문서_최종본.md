# 브랜치 전략 문서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 Git 브랜치 운영 기준을 정의한다.  
코드 안정성, 리뷰 품질, 배포 추적성, 핫픽스 대응을 위해 main, dev, feature, release, hotfix 브랜치를 구분해 운영한다.

## 2. 브랜치 구조

```text
main
├── release/v1.0.0
├── hotfix/v1.0.1-critical-login
└── dev
    ├── feature/123-payroll-plan-create
    ├── feature/124-daily-expense-create
    ├── fix/125-budget-remaining-recalculate
    ├── docs/126-api-spec-update
    └── refactor/127-payroll-service-cleanup
```

## 3. 브랜치 종류

| 브랜치       | 목적             | 생성 기준  | 병합 대상        | 보호      |
| ------------ | ---------------- | ---------- | ---------------- | --------- |
| `main`       | 운영 배포 기준   | 항상 존재  | 없음             | 최고 보호 |
| `dev`        | 통합 개발 기준   | 항상 존재  | main/release     | 보호      |
| `feature/*`  | 신규 기능        | Issue 기반 | dev              | 일반      |
| `fix/*`      | 버그 수정        | Issue 기반 | dev 또는 release | 일반      |
| `refactor/*` | 리팩터링         | Issue 기반 | dev              | 일반      |
| `docs/*`     | 문서 수정        | Issue 기반 | dev              | 일반      |
| `test/*`     | 테스트 추가      | Issue 기반 | dev              | 일반      |
| `release/*`  | 출시 후보 안정화 | 릴리즈 전  | main, dev        | 보호      |
| `hotfix/*`   | 운영 긴급 수정   | P0/P1 장애 | main, dev        | 보호      |

## 4. 브랜치명 규칙

```text
<type>/<issue-number>-<short-description>
```

예시:

```text
feature/123-payroll-plan-create
feature/124-community-post-create
fix/125-daily-budget-remaining
docs/126-readme-update
refactor/127-auth-token-service
test/128-payroll-calculation
hotfix/129-login-token-expiry
release/v1.0.0
```

## 5. type 값

| type     | 사용 시점                |
| -------- | ------------------------ |
| feature  | 신규 기능                |
| fix      | 버그 수정                |
| docs     | 문서 수정                |
| refactor | 기능 변화 없는 구조 개선 |
| test     | 테스트 추가/수정         |
| chore    | 빌드, 설정, 의존성       |
| ci       | CI/CD                    |
| hotfix   | 운영 긴급 수정           |
| release  | 출시 후보                |

## 6. main 브랜치 정책

| 항목       | 기준                         |
| ---------- | ---------------------------- |
| 직접 push  | 금지                         |
| 병합 방식  | Pull Request만 허용          |
| 승인       | Tech Lead 1명 이상 + QA 승인 |
| 테스트     | CI 전체 통과 필수            |
| 배포       | main 병합 후 운영 배포 후보  |
| 태그       | 릴리즈 시 `vX.Y.Z` 태그 생성 |
| force push | 금지                         |
| 삭제       | 금지                         |

## 7. dev 브랜치 정책

| 항목      | 기준                            |
| --------- | ------------------------------- |
| 직접 push | 금지                            |
| 병합 방식 | Pull Request                    |
| 승인      | 리뷰어 1명 이상                 |
| 테스트    | 기본 CI 통과                    |
| 용도      | 다음 릴리즈 후보 통합           |
| 배포      | dev/staging 환경 자동 배포 가능 |

## 8. feature 브랜치 정책

| 항목        | 기준                       |
| ----------- | -------------------------- |
| 생성 출발점 | 최신 dev                   |
| 브랜치 수명 | 해당 Issue 완료 시 삭제    |
| 커밋 단위   | 논리적 단위                |
| PR 크기     | 가능하면 500라인 이하 변경 |
| 병합 대상   | dev                        |
| 병합 후     | 브랜치 삭제                |

## 9. release 브랜치 정책

| 항목      | 기준                                 |
| --------- | ------------------------------------ |
| 생성 시점 | 출시 범위 freeze 후                  |
| 브랜치명  | `release/v1.0.0`                     |
| 허용 작업 | 버그 수정, 문서, 버전, 스토어 산출물 |
| 신규 기능 | 금지                                 |
| 병합 대상 | main, dev                            |
| 태그      | main 병합 후 `v1.0.0` 태그           |

## 10. hotfix 브랜치 정책

| 항목        | 기준                                             |
| ----------- | ------------------------------------------------ |
| 생성 출발점 | main                                             |
| 대상        | 운영 장애, 보안 취약점, 결제/인증/개인정보 P0/P1 |
| 브랜치명    | `hotfix/<issue>-<summary>`                       |
| 리뷰        | Tech Lead 필수                                   |
| QA          | 영향 범위 회귀 필수                              |
| 병합        | main 우선, 이후 dev 역병합                       |
| 릴리즈      | patch 버전 증가                                  |

## 11. Pull Request 병합 전략

| 상황               | 병합 방식    | 기준                |
| ------------------ | ------------ | ------------------- |
| 일반 feature → dev | Squash merge | 히스토리 단순화     |
| release → main     | Merge commit | 릴리즈 단위 보존    |
| hotfix → main      | Merge commit | 긴급 수정 이력 보존 |
| dev → release      | Merge commit | 통합 이력 보존      |
| docs 소규모 변경   | Squash merge | 단순화              |

## 12. PR 제목 규칙

```text
[type] #issue-number 작업 요약
```

예시:

```text
[feat] #123 급여 계획 생성 API 구현
[fix] #125 일일 예산 남은금액 재계산 오류 수정
[docs] #126 README 실행 방법 보완
```

## 13. 브랜치 보호 규칙

### main

- Require a pull request before merging
- Require approvals: 최소 2명
- Require status checks to pass
- Require branches to be up to date before merging
- Require conversation resolution before merging
- Restrict who can push
- Do not allow force pushes
- Do not allow deletions

### dev

- Require a pull request before merging
- Require approvals: 최소 1명
- Require status checks to pass
- Require conversation resolution before merging
- Do not allow force pushes

### release/\*

- Require a pull request before merging
- Require QA approval
- Require status checks
- Restrict force push/delete

## 14. 충돌 해결 기준

| 충돌 유형            | 책임                          |
| -------------------- | ----------------------------- |
| 기능 코드 충돌       | 해당 기능 담당자              |
| 공통 컴포넌트 충돌   | Frontend Lead                 |
| API 계약 충돌        | Backend Lead                  |
| DB 마이그레이션 충돌 | DB 변경 작성자 + Backend Lead |
| 문서 충돌            | 문서 변경자 + PM              |

## 15. 금지 사항

- main/dev 직접 push 금지
- Issue 번호 없는 브랜치 생성 금지
- 하나의 브랜치에서 여러 Epic 작업 금지
- release 브랜치에서 신규 기능 추가 금지
- hotfix 적용 후 dev 역병합 누락 금지
- force push로 리뷰 이력 손상 금지

## 16. 최종 완료 기준

본 브랜치 전략은 급여납치 플랫폼의 저장소 운영 기준으로 즉시 적용한다.  
브랜치 보호 규칙이 GitHub 저장소에 반영되고 모든 작업이 Issue 기반 PR로 병합될 때 본 문서의 적용이 완료된 것으로 본다.
