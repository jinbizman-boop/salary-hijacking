# Definition of Done 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 모든 작업이 “완료”로 인정되기 위한 공통 기준을 정의한다.  
완료란 단순히 코드가 작성된 상태가 아니라, 기능·테스트·보안·문서·리뷰·배포 가능성이 모두 검증된 상태를 의미한다.

## 2. 전역 완료 기준

모든 Issue는 아래 조건을 모두 만족해야 Done으로 전환할 수 있다.

| 번호 | 기준                                           | 필수 여부    |
| ---: | ---------------------------------------------- | ------------ |
|    1 | 요구사항과 수락 기준을 모두 충족한다           | 필수         |
|    2 | 관련 코드가 작성되고 로컬에서 정상 실행된다    | 필수         |
|    3 | 단위 테스트 또는 수동 검증 기록이 존재한다     | 필수         |
|    4 | TypeScript/린트/포맷 검사를 통과한다           | 필수         |
|    5 | Pull Request 리뷰가 완료되었다                 | 필수         |
|    6 | main/dev 브랜치 직접 푸시 없이 PR로 병합되었다 | 필수         |
|    7 | 관련 문서가 업데이트되었다                     | 필수         |
|    8 | 보안/개인정보 영향 검토가 완료되었다           | 해당 시 필수 |
|    9 | DB 변경 시 마이그레이션과 롤백이 작성되었다    | 해당 시 필수 |
|   10 | 배포 후 확인 항목이 정의되었다                 | 해당 시 필수 |

## 3. 기능 개발 DoD

| 항목      | 완료 기준                                |
| --------- | ---------------------------------------- |
| 기능 동작 | 정상 흐름과 주요 예외 흐름이 구현됨      |
| API 연동  | 요청/응답/오류 처리 기준 준수            |
| 상태관리  | 로딩, 성공, 빈 상태, 오류 상태 구현      |
| 입력 검증 | 필수값, 형식, 범위, 권한 검증            |
| 금액 처리 | 원 단위, 음수 금지, 콤마 표시, 초과 처리 |
| 접근성    | 터치 영역, 대비, 라벨, 포커스 흐름 확인  |
| 테스트    | 핵심 로직 테스트 또는 QA 케이스 연결     |
| 문서      | 기능 명세서 또는 API 문서 영향 반영      |

## 4. 프론트엔드 DoD

| 항목      | 기준                                          |
| --------- | --------------------------------------------- |
| 화면 구현 | 화면 설계서와 UI 구조 일치                    |
| 컴포넌트  | 디자인 시스템 컴포넌트 사용                   |
| 반응형    | 지정 디바이스 크기에서 깨짐 없음              |
| 상태      | loading, empty, error, success 상태 모두 구현 |
| 접근성    | 버튼 라벨, input label, semantic role 준수    |
| 성능      | 목록 렌더링, 이미지, 불필요한 re-render 점검  |
| 테스트    | 주요 컴포넌트 또는 화면 흐름 검증             |
| 스크린샷  | UI 변경 시 PR에 전/후 이미지 첨부             |

## 5. 백엔드 DoD

| 항목        | 기준                                               |
| ----------- | -------------------------------------------------- |
| API 계약    | API 명세서와 endpoint/method/body/status 일치      |
| 인증/인가   | 사용자 권한 검증 필수                              |
| DB 트랜잭션 | 금액/지출/저축/게시글 중요 변경은 트랜잭션 처리    |
| 입력 검증   | 서버 측 검증 필수                                  |
| 오류 응답   | 표준 errorCode와 requestId 포함                    |
| 로깅        | 실패 로그에 requestId, userId hash, errorCode 기록 |
| 테스트      | 성공/실패/권한/검증 실패 테스트 포함               |
| 성능        | 주요 쿼리 인덱스 사용 여부 확인                    |
| 보안        | 민감정보 로그 노출 금지                            |

## 6. DB 변경 DoD

| 항목         | 기준                                     |
| ------------ | ---------------------------------------- |
| 스키마       | 테이블 정의서와 일치                     |
| 마이그레이션 | forward migration 작성                   |
| 롤백         | rollback 절차 또는 보상 SQL 작성         |
| 기본값       | nullable/default/check constraint 검토   |
| 인덱스       | 조회 API에 필요한 인덱스 정의            |
| 데이터 보존  | 삭제/익명화 정책과 충돌 없음             |
| 샘플 데이터  | 개발/QA용 seed 필요 시 작성              |
| 검증         | 로컬 또는 dev DB에서 migration 실행 확인 |

## 7. 보안/개인정보 DoD

| 항목       | 기준                                          |
| ---------- | --------------------------------------------- |
| 민감정보   | 급여·대출·저축·소비 데이터 로그 노출 금지     |
| 토큰       | access/refresh token 저장·만료 기준 준수      |
| 권한       | 본인 데이터만 접근 가능                       |
| 파일       | 확장자, MIME, 크기, 악성 업로드 제한          |
| Rate Limit | 인증/글쓰기/댓글/파일 업로드에 제한 적용      |
| 광고 분리  | 재무성 개인정보를 광고 타겟팅에 사용하지 않음 |
| 동의       | 신규 수집/이용 목적 발생 시 동의 문서 반영    |

## 8. QA DoD

| 항목          | 기준                             |
| ------------- | -------------------------------- |
| 테스트 케이스 | 관련 테스트 케이스가 실행됨      |
| 회귀 테스트   | 기존 핵심 기능 영향 없음         |
| 버그          | P0/P1 미해결 버그 없음           |
| 금액 계산     | 계산 테스트 문서 기준 통과       |
| 디바이스      | 지정 대표 디바이스에서 확인      |
| UAT           | 출시 대상 기능이면 UAT 확인 완료 |

## 9. 문서 작업 DoD

| 항목      | 기준                                   |
| --------- | -------------------------------------- |
| 문서 형식 | Markdown 제목/표/목차 일관성 유지      |
| 용어      | 용어 정의서와 일치                     |
| 충돌 검토 | 기존 기획/API/DB/보안 문서와 충돌 없음 |
| 변경 이력 | 문서 버전 또는 변경 사유 기록          |
| 링크      | 관련 문서와 상호 참조                  |
| 최종 상태 | 문서상 실행 가능한 기준으로 작성됨     |

## 10. Pull Request 병합 전 체크리스트

```markdown
- [ ] 연결 Issue가 있다.
- [ ] 수락 기준을 모두 충족했다.
- [ ] 로컬 실행을 확인했다.
- [ ] 테스트를 실행했다.
- [ ] 린트/타입 검사를 통과했다.
- [ ] API/DB/보안/문서 영향 범위를 확인했다.
- [ ] 스크린샷 또는 로그 증빙을 첨부했다.
- [ ] 리뷰어 승인을 받았다.
- [ ] 배포 후 확인 항목을 작성했다.
```

## 11. Done 불가 조건

아래 중 하나라도 해당되면 Done으로 전환할 수 없다.

- 기능은 동작하지만 테스트가 없다.
- PR 리뷰 없이 병합되었다.
- API 명세와 실제 응답이 다르다.
- 금액 계산 결과가 테스트 문서와 다르다.
- 개인정보/급여 데이터가 로그에 노출된다.
- DB 변경이 있으나 마이그레이션 또는 롤백 계획이 없다.
- 운영/CS 영향이 있으나 운영 문서가 반영되지 않았다.
- 배포 후 확인 방법이 없다.

## 12. 최종 완료 선언

본 문서는 급여납치 플랫폼의 모든 작업 완료 기준으로 적용한다.  
Issue, PR, 스프린트, 릴리즈에서 “완료”라는 표현은 본 Definition of Done을 모두 충족한 상태만 의미한다.
