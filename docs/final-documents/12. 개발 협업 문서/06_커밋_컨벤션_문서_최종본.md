# 커밋 컨벤션 문서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 Git 커밋 메시지 작성 기준을 통일한다.  
커밋 로그만으로 변경 유형, 영향 범위, 연결 Issue, 배포 영향 여부를 파악할 수 있어야 한다.

## 2. 기본 형식

```text
<type>(<scope>): <subject>

<body>

<footer>
```

간단한 커밋은 한 줄로 작성할 수 있다.

```text
feat(payroll): 급여 계획 생성 API 추가
fix(budget): 일일 남은금액 재계산 오류 수정
docs(readme): 로컬 실행 방법 추가
```

## 3. type 정의

| type     | 의미        | 예시                        |
| -------- | ----------- | --------------------------- |
| feat     | 신규 기능   | 급여 계획 생성              |
| fix      | 버그 수정   | 지출 삭제 후 금액 오류 수정 |
| docs     | 문서 수정   | README 업데이트             |
| style    | 코드 스타일 | 포맷팅, 세미콜론            |
| refactor | 리팩터링    | 서비스 계층 분리            |
| test     | 테스트      | 유닛 테스트 추가            |
| chore    | 기타 작업   | 패키지 업데이트             |
| build    | 빌드 설정   | 번들 설정 변경              |
| ci       | CI/CD       | GitHub Actions 수정         |
| perf     | 성능 개선   | 목록 쿼리 최적화            |
| security | 보안 수정   | 토큰 검증 강화              |
| revert   | 되돌리기    | 이전 커밋 revert            |

## 5. subject 작성 규칙

| 기준   | 설명                            |
| ------ | ------------------------------- |
| 길이   | 72자 이내 권장                  |
| 문체   | 명령형 또는 완료형으로 간결하게 |
| 마침표 | 끝에 마침표 사용하지 않음       |
| 언어   | 한국어 사용                     |
| 내용   | 무엇을 변경했는지 명확히 작성   |

좋은 예:

```text
feat(payroll): 급여 계획 생성 API 추가
fix(expense): 변동지출 삭제 후 일일 사용금액 갱신
refactor(auth): 토큰 검증 로직 미들웨어로 분리
```

나쁜 예:

```text
수정
버그 고침
작업함
feat: 이것저것 추가
```

## 6. body 작성 기준

body는 변경 이유, 구현 방식, 주의사항을 설명한다.

```text
feat(payroll): 급여 계획 생성 API 추가

- 급여일, 수령 예정 급여, 목표 납치금액을 저장한다.
- 저장 후 예상 납치금액을 서버에서 계산해 반환한다.
- 음수 금액 입력 시 VALIDATION_ERROR를 반환한다.

Refs: #123
```

## 7. footer 작성 기준

| 항목            | 예시                                              |
| --------------- | ------------------------------------------------- |
| 연결 Issue      | `Refs: #123`                                      |
| 종료 Issue      | `Closes: #123`                                    |
| Breaking Change | `BREAKING CHANGE: payroll_plans.amount 컬럼 제거` |
| 보안 영향       | `Security: token rotation 적용`                   |
| DB 변경         | `Migration: 20260615_create_payroll_plans`        |

## 8. Breaking Change 기준

아래 변경은 반드시 `BREAKING CHANGE`를 footer에 적는다.

- API request/response 구조 변경
- 기존 DB 컬럼 제거/이름 변경
- 인증 방식 변경
- 클라이언트 업데이트 없이는 동작 불가능한 변경
- 환경변수 필수값 추가
- 기존 데이터 마이그레이션 필요

예시:

```text
feat(api): 급여 계획 응답 구조 변경

BREAKING CHANGE: expectedHijackAmount 필드가 hijackAmount로 변경됨
Refs: #211
```

## 9. Issue 연결 규칙

| 상황              | 사용               |
| ----------------- | ------------------ |
| 관련만 있음       | `Refs: #123`       |
| 병합 시 자동 종료 | `Closes: #123`     |
| 여러 Issue 관련   | `Refs: #123, #124` |
| 버그 해결         | `Fixes: #125`      |

## 10. 커밋 분리 기준

커밋은 아래 기준으로 분리한다.

| 분리 대상   | 예시              |
| ----------- | ----------------- |
| 기능 추가   | API 구현          |
| 테스트 추가 | API 테스트        |
| 문서 수정   | API 명세 업데이트 |
| DB 변경     | 마이그레이션      |
| 리팩터링    | 서비스 계층 분리  |
| 스타일      | 포맷팅            |

하나의 커밋에 기능, 문서, 리팩터링, 테스트를 모두 섞지 않는다.

## 11. 커밋 예시

### 기능

```text
feat(budget): 일일 예산 생성 기능 추가

- daily_budgets 테이블에 날짜별 예산을 저장한다.
- 사용금액과 남은금액을 서버 계산값으로 반환한다.

Closes: #141
```

### 버그

```text
fix(payroll): 급여보다 지출이 큰 경우 납치금액 음수 표시 방지

- 납치금액은 최소 0원으로 보정한다.
- 초과 지출 상태는 별도 flag로 반환한다.

Fixes: #182
```

### 문서

```text
docs(api): 변동지출 삭제 API 오류 코드 추가

Refs: #190
```

### 보안

```text
security(auth): refresh token 재사용 감지 처리 추가

- 재사용 감지 시 모든 세션을 만료한다.
- 보안 이벤트 로그를 남긴다.

Refs: #205
```

## 12. 금지 사항

- 의미 없는 메시지 금지: `update`, `fix`, `작업`, `수정`
- Issue 번호 누락 금지
- 민감정보를 커밋 메시지에 포함 금지
- API 키, DB URL, 토큰 커밋 금지
- 하나의 커밋에 무관한 변경 섞기 금지
- 리뷰 반영 커밋을 `fix`로 남발 금지. 필요 시 `refactor`, `style`, `test` 사용

## 13. 커밋 검증 기준

CI에서 아래 항목을 검증한다.

- type이 허용 목록에 있는가
- subject가 비어 있지 않은가
- 브랜치 또는 PR에 Issue 번호가 연결되어 있는가
- 커밋에 secret 패턴이 포함되지 않았는가
- Breaking Change가 있으면 릴리즈 노트 후보로 표시되는가

## 14. 최종 완료 기준

본 커밋 컨벤션은 급여납치 플랫폼의 모든 저장소에 적용한다.  
커밋 메시지가 본 기준을 만족하지 않으면 PR 리뷰에서 수정 요청 대상이 된다.
