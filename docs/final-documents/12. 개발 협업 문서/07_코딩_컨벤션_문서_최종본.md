# 코딩 컨벤션 문서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 프론트엔드, 백엔드, 공통 타입, 테스트, 문서화 코드 작성 기준을 통일하기 위한 최종 기준이다.  
코딩 컨벤션의 목적은 가독성, 유지보수성, 안정성, 보안성, 확장성을 확보하는 것이다.

## 2. 공통 원칙

| 원칙          | 기준                                             |
| ------------- | ------------------------------------------------ |
| 명확성        | 축약보다 의미가 분명한 이름을 사용한다           |
| 일관성        | 동일한 개념은 동일한 이름과 구조를 사용한다      |
| 타입 안정성   | any 사용을 금지하고 명시적 타입을 우선한다       |
| 단일 책임     | 함수/컴포넌트/서비스는 하나의 책임만 가진다      |
| 테스트 가능성 | 핵심 로직은 UI와 분리해 테스트 가능하게 작성한다 |
| 보안          | 민감정보는 로그, 상태, URL에 노출하지 않는다     |
| 문서 연계     | 기능 ID, 화면 ID, API ID와 연결 가능해야 한다    |

## 3. 네이밍 규칙

| 대상            | 규칙                        | 예시                      |
| --------------- | --------------------------- | ------------------------- |
| 파일            | kebab-case                  | `payroll-plan.service.ts` |
| 폴더            | kebab-case                  | `daily-budget`            |

## 4. 금액 변수 네이밍

급여납치는 금액 데이터가 핵심이므로 금액 변수는 아래 규칙을 따른다.

| 의미           | 변수명                  |
| -------------- | ----------------------- |
| 수령 예정 급여 | `expectedSalaryAmount`  |
| 지출 예정 금액 | `expectedExpenseAmount` |
| 납치금액       | `hijackAmount`          |
| 누적 납치금액  | `totalHijackAmount`     |
| 일일 설정금액  | `dailyLimitAmount`      |
| 일일 사용금액  | `dailyUsedAmount`       |
| 일일 남은금액  | `dailyRemainingAmount`  |
| 고정지출 금액  | `fixedExpenseAmount`    |
| 고정저축 금액  | `fixedSavingAmount`     |

금액 변수는 반드시 `Amount` suffix를 붙인다.

## 8. 함수 작성 규칙

| 항목   | 기준                                |
| ------ | ----------------------------------- |
| 길이   | 50라인 이하 권장                    |
| 입력   | 명시적 파라미터                     |
| 출력   | 명시적 반환 타입                    |
| 부작용 | 계산 함수는 부작용 금지             |
| 예외   | 예상 가능한 오류는 표준 에러로 처리 |

예시:

```ts
export function calculateHijackAmount(input: { expectedSalaryAmount: number; expectedExpenseAmount: number }): number {
  const result = input.expectedSalaryAmount - input.expectedExpenseAmount;
  return Math.max(0, result);
}
```

## 13. 주석 규칙

주석은 “무엇”이 아니라 “왜”를 설명한다.

좋은 예:

```ts
// 납치금액은 사용자에게 성과로 표시되므로 음수 대신 0원으로 고정한다.
return Math.max(0, salaryAmount - expenseAmount);
```

나쁜 예:

```ts
// salary에서 expense를 뺀다.
return salary - expense;
```
