# 급여납치 README 최종본

## 1. 프로젝트 소개

**급여납치**는 직장인과 아르바이트생이 월급을 받은 뒤 무계획 소비로 급여가 사라지는 문제를 줄이기 위해, 급여·고정지출·고정저축·생활비·변동지출을 한 화면에서 관리하고 남은 금액을 “납치금액”으로 시각화하는 모바일 급여관리/가계부 플랫폼이다.

서비스는 단순 지출 기록을 넘어 사용자가 월급을 받는 즉시 지켜야 할 돈을 먼저 분리하고, 일일 예산과 자기계발 루틴을 함께 관리하도록 설계된다.

## 2. 핵심 가치

| 가치            | 설명                                                  |
| --------------- | ----------------------------------------------------- |
| 월급 선점 관리  | 월급이 사라지기 전에 고정지출·저축·생활비를 먼저 분리 |
| 납치금액 시각화 | 사용자가 지켜낸 금액을 성과로 보여줌                  |
| 일일 소비 통제  | 오늘 사용할 수 있는 금액과 남은 금액을 즉시 확인      |
| 자기계발 루틴   | 독서, 뉴스, 영어, 건강 미션으로 재방문 유도           |
| 직장인 커뮤니티 | 인증과 정보 공유를 통해 지속 사용 동기 강화           |

## 6. 시작하기

### 6.1 저장소 클론

```bash
git clone https://github.com/<org>/salary-hijacking.git
cd salary-hijacking
```

### 6.2 패키지 설치

```bash
corepack enable
pnpm install
```

### 6.3 환경변수 설정

```bash
cp .env.example .env.local
```

`.env.local`에 로컬 개발용 값을 입력한다.

### 6.4 로컬 실행

```bash
pnpm dev
```

개별 실행:

```bash
pnpm --filter mobile dev
pnpm --filter api dev
```

## 10. 브랜치 전략

| 브랜치      | 목적           |
| ----------- | -------------- |
| main        | 운영 배포 기준 |
| dev         | 통합 개발 기준 |
| feature/\*  | 신규 기능      |
| fix/\*      | 버그 수정      |
| docs/\*     | 문서 수정      |
| refactor/\* | 리팩터링       |
| release/\*  | 출시 후보      |
| hotfix/\*   | 운영 긴급 수정 |

main/dev 직접 push는 금지한다. 모든 변경은 Pull Request를 통해 병합한다.

## 11. 커밋 컨벤션

```text
<type>(<scope>): <subject>
```

예시:

```text
feat(payroll): 급여 계획 생성 API 추가
fix(budget): 일일 남은금액 재계산 오류 수정
docs(readme): 개발 환경 세팅 방법 추가
```

허용 type:

```text
feat, fix, docs, style, refactor, test, chore, build, ci, perf, security, revert
```

## 12. Pull Request 기준

PR은 아래 조건을 충족해야 한다.

- 연결 Issue가 있다.
- 수락 기준이 작성되어 있다.
- 테스트 결과가 있다.
- 린트와 타입 검사를 통과했다.
- UI 변경 시 스크린샷이 있다.

- 보안/개인정보 영향이 검토되었다.
- 리뷰어 승인을 받았다.

## 14. 보안 주의사항

- `.env.local`을 커밋하지 않는다.
- API 키, DB URL, OAuth secret, FCM key를 코드에 직접 작성하지 않는다.
- 급여, 대출, 저축, 소비 내역을 로그에 남기지 않는다.
- 운영 데이터는 로컬 개발에 사용하지 않는다.
- 보안 이슈는 공개 Issue에 상세 재현 코드와 민감정보를 올리지 않는다.

## 15. 배포 개요

| 환경       | 목적        |
| ---------- | ----------- |
| local      | 개발자 로컬 |
| dev        | 통합 개발   |
| staging    | QA/UAT      |
| production | 운영        |

배포는 GitHub Actions와 Cloudflare 환경을 사용하며, main 병합 전 QA와 출시 체크리스트를 통과해야 한다.
