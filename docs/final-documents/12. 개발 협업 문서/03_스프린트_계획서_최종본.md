# 스프린트 계획서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼 개발을 반복 가능한 스프린트 단위로 운영하기 위한 최종 기준이다.  
스프린트는 2주 단위로 운영하며, MVP 기준 v1.0.0 출시까지의 개발 범위와 완료 기준을 고정한다.

## 2. 스프린트 운영 원칙

| 항목          | 기준                             |
| ------------- | -------------------------------- |
| 스프린트 길이 | 2주                              |
| 계획 회의     | 스프린트 시작일                  |
| 데일리 체크   | 매 영업일 15분 이내              |
| 중간 점검     | 스프린트 1주차 종료일            |
| 리뷰/데모     | 스프린트 종료일                  |
| 회고          | 리뷰 직후                        |
| 작업 단위     | GitHub Issue                     |
| 완료 기준     | Definition of Done 충족          |
| 변경 통제     | 스프린트 중 P0 외 범위 추가 금지 |

## 3. 역할과 책임

| 역할      | 책임                            |
| --------- | ------------------------------- |
| PO/PM     | 우선순위, 범위, 수락 기준 확정  |
| Tech Lead | 기술 설계, 리뷰 기준, 병합 승인 |
| Frontend  | 모바일 UI, 상태관리, API 연동   |
| Backend   | API, 인증, DB, 배치, 관리자 API |
| Designer  | UI 검수, 디자인 시스템 관리     |
| QA        | 테스트 케이스, 회귀, UAT 검증   |
| DevOps    | CI/CD, 환경변수, 배포, 모니터링 |
| Operator  | CS/공지/모더레이션/이벤트 검수  |

## 5. 스프린트 계획 회의 입력물

- 제품 요구사항 정의서
- 기능 명세서
- 화면 설계서
- API 명세서
- 데이터/DB 문서
- 보안/정책 문서
- QA 테스트 문서
- 이전 스프린트 미완료 항목
- 운영/CS 피드백

## 6. 스프린트 계획 산출물

| 산출물         | 설명                               |
| -------------- | ---------------------------------- |
| Sprint Goal    | 이번 스프린트의 하나의 명확한 목표 |
| Sprint Backlog | 선택된 Issue 목록                  |
| 담당자 배정표  | 작업별 담당자/리뷰어               |
| 위험 목록      | 차단 요인, 외부 의존성             |
| 완료 기준      | 수락 기준, 테스트 기준             |
| 데모 항목      | 스프린트 종료 시 보여줄 기능       |

## 7. 스프린트 중 변경 관리

| 상황              | 처리 방식                           |
| ----------------- | ----------------------------------- |
| P0 장애           | 즉시 스프린트에 포함                |
| P1 출시 차단 버그 | PM/Tech Lead 승인 후 포함           |
| 신규 기능 요청    | 다음 스프린트 백로그로 이동         |
| 디자인 변경       | 영향 분석 후 승인 필요              |
| DB 변경           | 마이그레이션 계획과 롤백 기준 필수  |
| 보안 변경         | 보안 요구사항 정의서 기준 우선 처리 |

## 8. 데일리 체크 양식

```text
어제 완료한 일:
오늘 진행할 일:
차단 요인:
리뷰 요청:
문서/QA 영향:
```

## 9. 스프린트 리뷰 기준

스프린트 리뷰에서는 아래 항목을 확인한다.

- Sprint Goal 달성 여부
- 완료된 Issue 수
- 미완료 Issue와 원인
- 주요 기능 데모
- QA 결과
- 성능/보안 이슈
- 다음 스프린트 이월 대상
- 문서 업데이트 여부

## 10. 스프린트 회고 기준

| 항목    | 질문                                      |
| ------- | ----------------------------------------- |
| Keep    | 계속 유지할 점은 무엇인가                 |
| Problem | 문제는 무엇인가                           |
| Try     | 다음 스프린트에서 시도할 개선은 무엇인가  |
| Risk    | 누적 위험은 무엇인가                      |
| Action  | 담당자와 기한이 있는 개선 작업은 무엇인가 |

## 11. 완료 기준

스프린트는 다음 조건을 만족할 때 종료한다.

- Sprint Goal 달성 여부가 기록되었다.
- Done Issue가 Definition of Done을 충족했다.
- 미완료 Issue의 이월 여부가 결정되었다.
- QA 결과가 기록되었다.
- 데모가 완료되었다.
- 회고 Action Item이 등록되었다.
