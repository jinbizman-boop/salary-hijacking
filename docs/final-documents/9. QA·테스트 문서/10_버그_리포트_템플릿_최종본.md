# 10. 버그 리포트 템플릿 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 오류를 일관된 형식으로 등록, 분류, 재현, 수정, 검증하기 위한 버그 리포트 템플릿과 처리 기준을 정의한다.

## 2. 버그 리포트 기본 템플릿

```markdown
# [BUG] 제목

- 버그 ID:
- 등록일:
- 등록자:
- 담당자:
- 상태: New / Assigned / In Progress / Fixed / Retest / Closed / Reopened / Deferred
- 심각도: Blocker / Critical / Major / Minor / Trivial
- 우선순위: P1 / P2 / P3
- 발견 환경: Local / Dev / Staging / Production
- 앱 버전:
- API 버전:
- OS/기기:
- 네트워크:

## 2. 요약

- 한 줄 요약:
- 영향 사용자:
- 영향 기능:
- 출시 차단 여부:

## 3. 재현 절차

1.
2.
3.
4.

## 4. 기대 결과

-

## 5. 실제 결과

-

## 6. 증거 자료

- 스크린샷:
- 영상:
- 로그:
- API requestId/traceId:
- 관련 DB 레코드 ID:

## 7. 추가 정보

- 발생 빈도: 항상 / 자주 / 가끔 / 1회
- 회귀 여부: 신규 / 회귀 / 기존 미발견
- 관련 요구사항/테스트 케이스:
```

## 3. 심각도 기준

| 심각도   | 정의                       | 예시                             | SLA            |
| -------- | -------------------------- | -------------------------------- | -------------- |
| Blocker  | 앱 사용 또는 출시 불가     | 로그인 전체 불가, 앱 실행 불가   | 즉시 대응      |
| Critical | 핵심 데이터/금액/보안 오류 | 납치금액 오계산, 타인 급여 노출  | 당일 수정 원칙 |
| Major    | 주요 기능 실패             | 지출 수정 실패, 게시글 등록 실패 | 다음 빌드 수정 |
| Minor    | 사용은 가능하나 품질 저하  | UI 깨짐, 문구 오류               | 계획 수정      |
| Trivial  | 영향 낮음                  | 내부 표기, 미세 정렬             | 백로그         |

## 4. 우선순위 기준

| 우선순위 | 기준                         | 출시 영향         |
| -------- | ---------------------------- | ----------------- |
| P1       | 핵심 기능/보안/금액/개인정보 | 출시 차단         |
| P2       | 주요 사용 흐름/운영 기능     | 출시 전 수정 원칙 |
| P3       | 편의/광고/비핵심 UI          | 차기 배포 가능    |

## 5. 상태 정의

| 상태        | 의미             | 다음 단계      |
| ----------- | ---------------- | -------------- |
| New         | 신규 등록        | triage 필요    |
| Assigned    | 담당자 배정      | 원인 분석      |
| In Progress | 수정 중          | PR 생성        |
| Fixed       | 수정 완료        | QA 재검증      |
| Retest      | 재검증 중        | PASS/FAIL 판정 |
| Closed      | 종료             | 회귀 범위 반영 |
| Reopened    | 재발 또는 미수정 | 재수정         |
| Deferred    | 승인된 연기      | 백로그 관리    |

## 6. 필수 첨부 기준

| 결함 유형      | 필수 증거                              |
| -------------- | -------------------------------------- |
| UI 결함        | 스크린샷, 기기 정보                    |
| API 결함       | request/response, status code, traceId |
| 금액 계산 결함 | 입력값, 기대 계산식, 실제 결과         |
| 보안 결함      | payload, 요청 헤더, 영향 범위          |
| 성능 결함      | 측정 시간, p95/p99, 로그               |
| 크래시         | crash log, OS/기기, 재현 영상          |

## 7. 좋은 버그 제목 규칙

| 나쁜 제목   | 좋은 제목                                                         |
| ----------- | ----------------------------------------------------------------- |
| 오류남      | [급여 홈] 변동지출 삭제 후 남은금액이 재계산되지 않음             |
| 로그인 안됨 | [인증] 만료 refresh token 사용 시 재로그인 화면으로 이동하지 않음 |
| 글쓰기 이상 | [커뮤니티] 익명 글 작성 후 작성자 닉네임이 목록에 노출됨          |

## 8. 버그 중복 판단 기준

1. 동일 화면, 동일 원인, 동일 기대결과 불일치이면 중복으로 처리한다.
2. 동일 증상이지만 원인이 다른 경우 별도 버그로 등록한다.
3. 서로 다른 OS에서만 발생하는 경우 동일 버그에 환경 정보를 추가하거나 별도 하위 이슈로 관리한다.
4. 보안 결함은 중복 여부와 관계없이 영향 범위를 별도 기록한다.

## 9. 재검증 기준

| 항목      | 기준                             |
| --------- | -------------------------------- |
| 수정 빌드 | 수정 커밋이 포함된 Staging 빌드  |
| 원 결함   | 동일 절차로 재현되지 않아야 함   |
| 회귀 범위 | 관련 기능 최소 3개 이상 확인     |
| 로그      | 동일 오류 로그 재발 없음         |
| 최종 상태 | PASS 시 Closed, 실패 시 Reopened |

## 10. 최종 완료 기준

본 버그 리포트 템플릿은 급여납치 플랫폼의 결함 관리에 필요한 모든 입력 필드, 상태, 심각도, 우선순위, 재검증 기준을 포함한다. 모든 결함은 본 템플릿을 기준으로 등록한다.
