# 06. 금액 데이터 처리 정책서 최종본

> 문서 상태: 최종본  
> 적용 서비스: 급여납치(Salary Hijacking) 모바일 플랫폼  
> 적용 범위: 급여관리 · 가계부 · 자기계발 레벨업 · 커뮤니티 · 알림 · 광고/제휴 · 운영 데이터  
> 기준 DB: Neon DB / PostgreSQL  
> 기준일: 2026-06-15  
> 원칙: 본 문서는 문서상·이론상 추가 작업 없이 구현 기준으로 사용할 수 있도록 완결된 기준값, 필드, 관계, 정책, 예외를 포함한다.

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 모든 금액 데이터 저장, 계산, 표시, 검증, 초과 처리, 정정 처리 기준을 정의한다. 급여·지출·저축·예산·납치금액 계산 오류를 방지하는 것을 최우선 목적으로 한다.

| 원칙      | 기준                                                        |

## 3. 금액 필드 분류

| 분류      | 필드                     | 의미                             | 계산 포함 기준      |
| --------- | ------------------------ | -------------------------------- | ------------------- |
| 급여      | expected_salary_amount   | 이번 달 수령 예정 급여           | 급여 계획 기준값    |
| 지출 예정 | expected_expense_amount  | 이번 달 예상 소비성 지출         | 예상 납치금액 차감  |
| 목표      | target_hijack_amount     | 사용자가 지키고 싶은 목표 금액   | 달성률 계산         |
| 예상 납치 | expected_hijack_amount   | 급여에서 예상 지출을 제외한 금액 | 홈/계획 화면 표시   |
| 확정 납치 | confirmed_hijack_amount  | 월말 확정 보호 금액              | 누적 납치금액 포함  |
| 고정지출  | fixed_expenses.amount    | 반복 소비성 지출                 | 지출 합계 포함      |
| 고정저축  | savings_plans.amount     | 보호/저축 자금                   | 확정 납치 후보      |
| 일일 한도 | daily_limit_amount       | 하루 생활비 예산                 | 잔여/초과 계산 기준 |
| 사용금액  | used_amount              | 하루 활성 변동지출 합계          | 일일예산 계산       |
| 남은금액  | remaining_amount         | 하루 남은 예산                   | 화면 표시           |
| 초과금액  | over_amount              | 하루 예산 초과분                 | 경고/알림 기준      |
| 변동지출  | variable_expenses.amount | 일회성 소비                      | used_amount 합산    |

## 4. 저장 정책

### 4.1 허용 입력

| 입력 예   |    저장값 | 허용 여부                    | 설명                                    |
| --------- | --------: | ---------------------------- | --------------------------------------- |
| 2700000   |   2700000 | 허용                         | 숫자 입력                               |
| 2,700,000 |   2700000 | 허용                         | 콤마 제거 후 저장                       |
| 0         |         0 | 조건부 허용                  | 급여/예산/목표는 허용, 지출 항목은 불가 |
| -1000     | 저장 불가 | 불가                         | 음수 금지                               |
| 1000.5    | 저장 불가 | 불가                         | 소수점 금지                             |
| 1만원     |     10000 | 클라이언트 파싱 후 허용 가능 | 정규화 필요                             |
| abc       | 저장 불가 | 불가                         | 숫자 아님                               |

### 4.2 DB 저장 형식

```json
{
  "expected_salary_amount": 2700000,
  "daily_limit_amount": 20000,
  "used_amount": 13000,
  "remaining_amount": 7000,
  "over_amount": 0
}
```

### 4.3 화면 표시 형식

|           저장값 | 화면 표시    |
| ---------------: | ------------ |
|          2700000 | 2,700,000원  |
|            20000 | 20,000원     |
|                0 | 0원          |
| over_amount=3000 | 3,000원 초과 |

## 5. 핵심 계산식

### 5.1 예상 납치금액

```text
expected_hijack_amount = max(expected_salary_amount - expected_expense_amount, 0)
```

| 항목           |        예시 |
| -------------- | ----------: |
| 수령 예정 급여 | 2,700,000원 |
| 지출 예정 금액 |   600,000원 |
| 예상 납치금액  | 2,100,000원 |

### 5.2 일일 사용금액

```text
used_amount = sum(variable_expenses.amount where status = 'ACTIVE')
```

삭제, 환불, 정정 상태의 변동지출은 합산하지 않는다.

### 5.3 일일 남은금액

```text
remaining_amount = max(daily_limit_amount - used_amount, 0)
```

남은금액은 음수로 저장하지 않는다. 초과 사용은 `over_amount`에 저장한다.

### 5.4 일일 초과금액

```text
over_amount = max(used_amount - daily_limit_amount, 0)
```

### 5.5 일일예산 상태

```text
if used_amount > daily_limit_amount:
    status = 'EXCEEDED'
elif used_amount >= daily_limit_amount * 0.8:
    status = 'WARNING'
else:
    status = 'NORMAL'
```

### 5.6 월 소비성 지출 합계

```text
monthly_consumption_amount =
  sum(fixed_expenses.amount where status in ('SCHEDULED','PAID'))
  + sum(variable_expenses.amount where status = 'ACTIVE' and spent_at in month)
```

### 5.7 월 확정 저축 합계

```text
monthly_confirmed_savings_amount =
  sum(savings_plans.amount where status = 'TRANSFERRED')
```

### 5.8 월말 확정 납치금액

급여납치 플랫폼에서 확정 납치금액은 월급 중 소비되지 않고 사용자가 확보한 금액이다.

```text
confirmed_hijack_amount = max(
  expected_salary_amount - actual_consumption_amount,
  0
)
```

여기서 `actual_consumption_amount`는 소비성 지출만 포함한다. 고정저축은 소비가 아니라 확보 자금으로 간주한다.

### 5.9 누적 납치금액

```text
total_hijack_amount = sum(payroll_plans.confirmed_hijack_amount where status = 'CLOSED')
```

### 5.10 목표 달성률

```text
achievement_rate =
  if target_hijack_amount = 0 then 0
  else min(floor(confirmed_hijack_amount / target_hijack_amount * 100), 999)
```

화면에서는 `%` 단위로 표시한다. 내부 저장은 필요 시 정수로 저장한다.

## 6. 초과 처리 정책

| 상황                            | 처리 기준                                      | 알림                     |
| ------------------------------- | ---------------------------------------------- | ------------------------ |
| 사용금액이 일일 예산의 80% 이상 | status=WARNING                                 | 예산 주의 알림 생성 가능 |
| 사용금액이 일일 예산 초과       | status=EXCEEDED, remaining=0, over_amount 계산 | 예산 초과 알림 생성      |
| 지출 추가 후 예산 초과          | 지출은 저장, 예산 상태 갱신                    | 즉시 토스트 + 알림       |
| 예산 0원 상태에서 지출          | used_amount=지출, remaining=0, over=지출       | 초과 상태                |

## 7. 환불/삭제/정정 정책

### 7.1 삭제

| 항목               | 처리                                                     |
| ------------------ | -------------------------------------------------------- |
| 사용자가 지출 삭제 | `variable_expenses.status='DELETED'`, `deleted_at=now()` |
| 예산 재계산        | 삭제된 지출 제외                                         |
| 감사 로그          | 삭제 전/후 기록                                          |

### 7.2 환불

| 항목      | 처리                                                                      |
| --------- | ------------------------------------------------------------------------- |
| 전체 환불 | 원 지출 `status='REFUNDED'`                                               |
| 부분 환불 | 원 지출은 유지하고 환불 전용 정정 레코드 또는 `ADJUSTED` 대체 레코드 생성 |
| 음수 저장 | 금지                                                                      |

### 7.3 정정

| 항목             | 처리                                                                  |
| ---------------- | --------------------------------------------------------------------- |
| 금액 오입력 수정 | 기존 레코드 `ADJUSTED`, 새 ACTIVE 레코드 생성 또는 업데이트 방식 사용 |
| 변경 이력        | `audit_logs`에 before/after 저장                                      |
| 재계산           | 수정 완료 후 DailyBudget 재계산                                       |

## 8. 동시성 처리

지출을 동시에 여러 번 추가하는 경우 금액 불일치를 방지하기 위해 다음 중 하나를 적용한다.

1. 서버 트랜잭션 내부에서 지출 생성과 예산 재계산을 함께 수행한다.
2. `daily_budgets` 행에 대해 `SELECT ... FOR UPDATE`를 사용한다.
3. 이벤트 기반 처리 시 최종 정합성 보정 배치를 운영한다.

기본 구현 기준은 1번과 2번을 병행한다.

## 11. 최종 적용 기준

금액 처리 정책은 DB 제약조건, API validation, 클라이언트 입력 제한, QA 테스트가 모두 동일하게 따라야 한다. 금액 관련 정책은 사용자 신뢰와 직결되므로 본 문서와 다른 예외 처리를 허용하지 않는다.
