# 07. 데이터 보존/삭제 정책서 최종본

> 문서 상태: 최종본  
> 적용 서비스: 급여납치(Salary Hijacking) 모바일 플랫폼  
> 적용 범위: 급여관리 · 가계부 · 자기계발 레벨업 · 커뮤니티 · 알림 · 광고/제휴 · 운영 데이터  
> 기준 DB: Neon DB / PostgreSQL  
> 기준일: 2026-06-15  
> 원칙: 본 문서는 문서상·이론상 추가 작업 없이 구현 기준으로 사용할 수 있도록 완결된 기준값, 필드, 관계, 정책, 예외를 포함한다.

## 1. 문서 목적

본 문서는 급여납치 플랫폼에서 수집·생성되는 사용자 데이터의 보존, 삭제, 익명화, 백업, 탈퇴, 커뮤니티 글 처리 기준을 정의한다. 개인정보와 민감 재무 데이터를 안전하게 관리하면서 서비스 통계와 운영 안정성을 확보하는 것을 목적으로 한다.

## 2. 정책 원칙

| 원칙                | 정의                                                                 |
| ------------------- | -------------------------------------------------------------------- |
| 최소 보존           | 서비스 제공에 필요한 기간만 개인 데이터를 보존한다.                  |
| 목적 분리           | 급여/지출/저축 원천 데이터와 광고/통계 데이터를 분리한다.            |
| 탈퇴권 보장         | 사용자는 계정 탈퇴와 개인 데이터 삭제를 요청할 수 있다.              |
| 익명화 우선         | 커뮤니티 맥락 보존이 필요한 경우 작성자 개인정보를 익명화한다.       |
| 복구 가능 기간 명시 | 실수 탈퇴/장애 대응을 위한 제한적 유예 기간을 둔다.                  |
| 백업 삭제 지연 고지 | 백업 데이터는 운영 백업 주기에 따라 순차 삭제된다.                   |

## 4. 회원 탈퇴 처리 기준

### 4.1 탈퇴 요청 단계

1. 사용자가 마이페이지 또는 계정 설정에서 탈퇴를 요청한다.
2. 탈퇴 전 안내 화면에서 삭제 범위와 복구 가능 기간을 고지한다.
3. 사용자가 최종 확인을 완료하면 `users.status='WITHDRAWN'`으로 전환한다.
4. 로그인 세션과 푸시 토큰을 폐기한다.
5. 개인정보 삭제/익명화 작업을 큐에 등록한다.

### 4.2 탈퇴 유예 기간

| 항목                     | 기준                      |
| ------------------------ | ------------------------- |
| 기본 유예 기간           | 7일                       |
| 유예 기간 중 로그인      | 복구 확인 화면 표시       |
| 유예 기간 중 데이터 노출 | 일반 서비스 이용 불가     |
| 유예 기간 종료 후        | 개인정보 삭제/익명화 실행 |

### 4.3 탈퇴 후 삭제 대상

| 대상                | 처리                              |
| ------------------- | --------------------------------- |
| 이메일              | null 또는 irrecoverable hash 처리 |
| 전화번호            | null 처리                         |
| 소셜 provider key   | 삭제 또는 revoked 처리            |
| 프로필 이름/이미지  | 삭제 또는 익명 사용자로 치환      |
| 급여/지출/저축/예산 | 물리 삭제 또는 암호화 파기        |
| 알림                | 삭제                              |
| 레벨업 활동         | 삭제                              |
| 좋아요/북마크       | 삭제                              |
| 광고 로그           | user_id null 처리                 |

## 5. 커뮤니티 글 처리 정책

커뮤니티는 다른 사용자 댓글과 문맥이 연결되므로, 탈퇴 시 모든 글을 무조건 물리 삭제하면 대화 흐름이 깨질 수 있다. 따라서 다음 기준을 적용한다.

| 요청/상황             | 게시글 처리                     | 댓글 처리               | 작성자 표시             |
| --------------------- | ------------------------------- | ----------------------- | ----------------------- |
| 단순 탈퇴             | 게시글 유지 가능                | 댓글 유지 가능          | 탈퇴한 사용자 또는 익명 |
| 사용자가 글 삭제 요청 | `status='DELETED'`, 본문 마스킹 | 댓글은 정책에 따라 유지 | 표시 안 함              |
| 개인정보 포함 글      | 즉시 삭제/마스킹                | 관련 댓글 검토          | 표시 안 함              |
| 신고로 차단           | `status='BLOCKED'`              | 필요 시 댓글 숨김       | 운영 정책 적용          |
| 법적/고위험 사안      | 운영 보존                       | 운영 보존               | 외부 노출 차단          |

### 5.1 익명화 기준

| 필드              | 처리                                                     |
| ----------------- | -------------------------------------------------------- |
| user_id           | 내부 운영 필요 시 보관 후 별도 익명화 테이블로 분리 가능 |
| display_name      | “탈퇴한 사용자” 또는 “익명”으로 표시                     |
| profile_image_url | null 처리                                                |
| 게시글 본문       | 사용자가 삭제 요청한 경우 마스킹                         |
| 댓글 본문         | 사용자가 삭제 요청한 경우 “삭제된 댓글입니다.” 표시      |

## 6. 백업 데이터 보존/삭제

| 백업 유형          | 보존 기준           | 삭제 기준                          |
| ------------------ | ------------------- | ---------------------------------- |
| 일일 DB 백업       | 7~30일              | 보존 기간 만료 시 자동 삭제        |
| 주간 백업          | 4~12주              | 보존 기간 만료 시 자동 삭제        |
| 장애 복구 스냅샷   | 장애 대응 목적 기간 | 복구 완료 후 정책에 따라 삭제      |
| 파일 스토리지 백업 | 객체 보존 기간 연동 | 원본 삭제 후 백업 주기에 따라 삭제 |

백업에는 탈퇴 직후 삭제된 데이터가 일시적으로 포함될 수 있다. 단, 백업은 일반 서비스 조회에 사용하지 않으며 복구 목적 외 접근을 제한한다.

## 7. 데이터 삭제 방식

| 방식          | 적용 대상                                  | 설명                              |
| ------------- | ------------------------------------------ | --------------------------------- |
| Soft Delete   | 게시글, 댓글, 캠페인, 일부 운영 데이터     | `deleted_at`, `status`로 비활성화 |
| Hard Delete   | 급여, 지출, 저축, 예산, 알림               | 물리 삭제 또는 암호화 파기        |
| Anonymization | 커뮤니티 작성자, 광고 로그, 감사 로그 일부 | 개인 식별 불가능하게 치환         |
| Masking       | 삭제된 댓글/게시글 본문                    | “삭제된 내용입니다.” 표시         |
| Nullification | 광고 로그 user_id                          | user_id를 null 처리               |

## 8. 삭제 프로세스

```text
탈퇴 요청
→ 사용자 상태 WITHDRAWN
→ 세션/푸시 토큰 폐기
→ 삭제 작업 큐 등록
→ 재무 데이터 삭제
→ 알림/설정/반응 삭제
→ 커뮤니티 작성자 익명화 또는 삭제 요청 반영
→ 광고 로그 user_id null
→ 감사 로그 최소화
→ 사용자 상태 DELETED
→ 삭제 완료 알림 또는 이메일 발송 생략/처리 기록 저장
```

## 9. 데이터 보존/삭제 감사 로그

탈퇴 및 삭제 작업은 다음 로그를 남긴다.

| 필드          | 예시                                       |
| ------------- | ------------------------------------------ |
| action        | USER_WITHDRAW_REQUESTED, USER_DATA_DELETED |
| actor_user_id | 탈퇴 사용자 ID                             |
| target_type   | USER                                       |
| target_id     | user_id                                    |
| before_data   | 삭제 전 상태 최소 정보                     |
| after_data    | 삭제 후 상태 최소 정보                     |
| created_at    | 처리 시각                                  |

감사 로그에는 급여액, 지출 상세, 저축 상세를 저장하지 않는다.

## 12. 최종 적용 기준

본 정책은 앱 약관, 개인정보처리방침, DB 삭제 배치, 관리자 도구, CS 대응 매뉴얼과 일치해야 한다. 급여·지출·저축 정보는 서비스 신뢰의 핵심이므로 탈퇴 후 일반 운영 조회 및 광고 활용을 허용하지 않는다.
