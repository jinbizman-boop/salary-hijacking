# 01. 데이터 모델 정의서 최종본

> 문서 상태: 최종본  
> 적용 서비스: 급여납치(Salary Hijacking) 모바일 플랫폼  
> 적용 범위: 급여관리 · 가계부 · 자기계발 레벨업 · 커뮤니티 · 알림 · 광고/제휴 · 운영 데이터  
> 기준 DB: Neon DB / PostgreSQL  
> 기준일: 2026-06-15  
> 원칙: 본 문서는 문서상·이론상 추가 작업 없이 구현 기준으로 사용할 수 있도록 완결된 기준값, 필드, 관계, 정책, 예외를 포함한다.

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 전체 데이터 구조를 정의한다. 앱 화면, 기능 명세, 계산 정책, 알림, 레벨업, 커뮤니티, 광고/제휴, 운영 로그가 동일한 데이터 기준으로 구현되도록 모든 핵심 객체와 관계를 고정한다.

## 2. 데이터 모델 설계 원칙

| 원칙                 | 정의                                                                  | 적용 기준                                                            |
| -------------------- | --------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 사용자 중심 소유권   | 모든 개인 데이터는 `user_id`로 귀속된다.                              | 급여, 지출, 저축, 예산, 미션, 알림, 게시글 작성자                    |
| 금액 정수 저장       | 원화 금액은 소수점 없이 정수로 저장한다.                              | `BIGINT`, 단위 KRW 원                                                |
| 계획/실적 분리       | 사용자가 세운 계획과 실제 소비 기록은 분리한다.                       | PayrollPlan, FixedExpense, SavingsPlan, DailyBudget, VariableExpense |
| 소비/저축 구분       | 소비성 지출과 보호/확보 자금인 저축은 분리한다.                       | FixedExpense와 SavingsPlan 분리                                      |
| 누적성과 계산 가능성 | 월별 납치금액과 누적 납치금액을 재계산 가능하게 한다.                 | 월별 계획, 일별 예산, 변동지출 이력 보존                             |
| 익명성/운영성 병행   | 커뮤니티 익명 표시는 지원하되 내부 운영 식별자는 보존한다.            | CommunityPost의 `is_anonymous`, 내부 `user_id`                       |
| 민감정보 최소화      | 급여·대출·소비 등 민감 데이터는 목적별 최소 필드만 저장한다.          | 광고 데이터와 분리                                                   |
| 확장성               | MVP 이후 레벨업, 커뮤니티, 광고를 추가해도 핵심 모델이 깨지지 않는다. | 도메인별 독립 테이블                                                 |

## 3. 도메인 구분

| 도메인    | 설명                                             | 대표 객체                                                           |
| --------- | ------------------------------------------------ | ------------------------------------------------------------------- |
| 회원/인증 | 사용자 계정, 소셜 로그인, 프로필, 앱 설정        | User, AuthIdentity, UserProfile, UserSetting                        |
| 급여/계획 | 급여일, 수령 예정 급여, 지출 예정 금액, 목표금액 | PayrollPlan                                                         |
| 고정지출  | 반복 결제, 구독료, 대출금, 보험료, 통신비        | FixedExpense                                                        |
| 고정저축  | 청약, 적금, 투자, 자동이체 등 보호성 자금        | SavingsPlan                                                         |
| 일일예산  | 하루 생활비 한도, 사용금액, 잔여금액, 초과금액   | DailyBudget                                                         |
| 변동지출  | 커피, 식사, 쇼핑, 교통 등 당일 소비 기록         | VariableExpense                                                     |
| 알림      | 예산 초과, 결제 예정, 목표 달성, 레벨업, 이벤트  | Notification                                                        |
| 레벨업    | 독서, 뉴스, 영어, 건강 미션과 경험치/레벨        | GrowthTask, GrowthTaskCompletion, UserGrowthStat                    |
| 커뮤니티  | 게시글, 댓글, 좋아요, 조회수, 공유, 신고         | CommunityPost, CommunityComment, CommunityReaction, CommunityReport |
| 광고/제휴 | 배너 캠페인, 노출 위치, 클릭/노출 로그           | AdCampaign, AdImpression                                            |
| 파일/첨부 | 게시글, 프로필, 콘텐츠 첨부 파일                 | Attachment                                                          |
| 감사/운영 | 중요한 변경 이력, 삭제, 신고, 관리자 조치        | AuditLog                                                            |

## 4. 핵심 객체 정의

### 4.1 User

| 항목      | 정의                                                                                                  |
| --------- | ----------------------------------------------------------------------------------------------------- |
| 객체명    | User                                                                                                  |
| 목적      | 급여납치 플랫폼의 회원 계정 원장                                                                      |
| 주요 필드 | `user_id`, `email`, `phone_number`, `nickname`, `status`, `created_at`, `last_login_at`, `deleted_at` |
| 소유 관계 | 모든 개인 도메인의 부모 객체                                                                          |
| 민감도    | 개인정보                                                                                              |
| 필수 여부 | 필수                                                                                                  |
| 삭제 기준 | 탈퇴 시 개인정보 삭제 또는 익명화, 서비스 통계용 비식별 집계만 유지                                   |

### 4.2 AuthIdentity

| 항목      | 정의                                                                                          |
| --------- | --------------------------------------------------------------------------------------------- |
| 객체명    | AuthIdentity                                                                                  |
| 목적      | 이메일/네이버/카카오/구글/페이스북 등 로그인 수단 관리                                        |
| 주요 필드 | `identity_id`, `user_id`, `provider`, `provider_user_key`, `email`, `linked_at`, `revoked_at` |
| 관계      | User 1:N AuthIdentity                                                                         |
| 정책      | 한 사용자는 여러 로그인 수단을 연결할 수 있다. 동일 provider/user key는 중복될 수 없다.       |

### 4.3 UserProfile

| 항목      | 정의                                                                                                         |
| --------- | ------------------------------------------------------------------------------------------------------------ |
| 객체명    | UserProfile                                                                                                  |
| 목적      | 앱 표시용 프로필 관리                                                                                        |
| 주요 필드 | `profile_id`, `user_id`, `display_name`, `job_title`, `profile_image_url`, `bio`, `created_at`, `updated_at` |
| 관계      | User 1:1 UserProfile                                                                                         |
| 정책      | 커뮤니티 익명 글에는 `display_name`을 표시하지 않는다.                                                       |

### 4.4 UserSetting

| 항목      | 정의                                                                                                                                    |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| 객체명    | UserSetting                                                                                                                             |
| 목적      | 알림, 자동로그인, 마케팅 수신, 표시 단위 등 사용자 설정 관리                                                                            |
| 주요 필드 | `setting_id`, `user_id`, `push_enabled`, `budget_alert_enabled`, `growth_alert_enabled`, `marketing_opt_in`, `created_at`, `updated_at` |
| 관계      | User 1:1 UserSetting                                                                                                                    |
| 정책      | 필수 금융/보안 알림과 마케팅/콘텐츠 알림은 분리한다.                                                                                    |

### 4.5 PayrollPlan

| 항목      | 정의                                                                                                                                                                  |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 객체명    | PayrollPlan                                                                                                                                                           |
| 목적      | 급여일, 수령 예정 급여, 지출 예정 금액, 목표 납치금액 관리                                                                                                            |
| 주요 필드 | `payroll_plan_id`, `user_id`, `year_month`, `payday`, `expected_salary_amount`, `expected_expense_amount`, `target_hijack_amount`, `expected_hijack_amount`, `status` |
| 관계      | User 1:N PayrollPlan                                                                                                                                                  |
| 정책      | 사용자별 동일 `year_month` 활성 계획은 1개만 허용한다.                                                                                                                |
| 계산      | `expected_hijack_amount = max(expected_salary_amount - expected_expense_amount, 0)`                                                                                   |

### 4.6 FixedExpense

| 항목      | 정의                                                                                                                                  |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 객체명    | FixedExpense                                                                                                                          |
| 목적      | 구독료, 대출, 보험, 통신비 등 반복 소비성 지출 관리                                                                                   |
| 주요 필드 | `fixed_expense_id`, `user_id`, `payroll_plan_id`, `expense_day`, `category`, `name`, `amount`, `recurrence_type`, `status`, `paid_at` |
| 관계      | PayrollPlan 1:N FixedExpense                                                                                                          |
| 정책      | 고정지출은 소비성 지출로 계산되며 납치금액에서 차감한다.                                                                              |

### 4.7 SavingsPlan

| 항목      | 정의                                                                                                                                       |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| 객체명    | SavingsPlan                                                                                                                                |
| 목적      | 청약, 적금, 투자, 자동이체 등 보호성 자금 관리                                                                                             |
| 주요 필드 | `savings_plan_id`, `user_id`, `payroll_plan_id`, `saving_day`, `category`, `name`, `amount`, `recurrence_type`, `status`, `transferred_at` |
| 관계      | PayrollPlan 1:N SavingsPlan                                                                                                                |
| 정책      | 고정저축은 소비성 지출이 아니라 확정 납치금액 후보로 분류한다.                                                                             |

### 4.8 DailyBudget

| 항목      | 정의                                                                                                                          |
| --------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 객체명    | DailyBudget                                                                                                                   |
| 목적      | 일일 생활비 한도와 사용/잔여/초과 금액 관리                                                                                   |
| 주요 필드 | `daily_budget_id`, `user_id`, `budget_date`, `daily_limit_amount`, `used_amount`, `remaining_amount`, `over_amount`, `status` |
| 관계      | User 1:N DailyBudget                                                                                                          |
| 정책      | 사용자별 동일 날짜의 예산은 1개만 허용한다.                                                                                   |
| 계산      | `remaining_amount = max(daily_limit_amount - used_amount, 0)`, `over_amount = max(used_amount - daily_limit_amount, 0)`       |

### 4.9 VariableExpense

| 항목      | 정의                                                                                                                     |
| --------- | ------------------------------------------------------------------------------------------------------------------------ |
| 객체명    | VariableExpense                                                                                                          |
| 목적      | 일회성 소비 기록 관리                                                                                                    |
| 주요 필드 | `variable_expense_id`, `user_id`, `daily_budget_id`, `spent_at`, `category`, `merchant_name`, `memo`, `amount`, `status` |
| 관계      | DailyBudget 1:N VariableExpense                                                                                          |
| 정책      | 사용자가 소비를 추가/수정/삭제하면 연결된 DailyBudget 금액을 재계산한다.                                                 |

### 4.10 Notification

| 항목      | 정의                                                                                                                   |
| --------- | ---------------------------------------------------------------------------------------------------------------------- |
| 객체명    | Notification                                                                                                           |
| 목적      | 앱 내 알림 및 푸시 알림 원장                                                                                           |
| 주요 필드 | `notification_id`, `user_id`, `type`, `title`, `body`, `target_screen`, `status`, `scheduled_at`, `sent_at`, `read_at` |
| 관계      | User 1:N Notification                                                                                                  |
| 정책      | 사용자 설정에서 끈 콘텐츠 알림은 생성하지 않거나 발송하지 않는다.                                                      |

### 4.11 GrowthTask

| 항목      | 정의                                                                                                 |
| --------- | ---------------------------------------------------------------------------------------------------- |
| 객체명    | GrowthTask                                                                                           |
| 목적      | 독서, 뉴스, 영어, 건강 미션 원장                                                                     |
| 주요 필드 | `growth_task_id`, `type`, `title`, `description`, `exp_reward`, `active_from`, `active_to`, `status` |
| 관계      | GrowthTask 1:N GrowthTaskCompletion                                                                  |
| 정책      | 미션 완료 시 경험치와 레벨 통계가 증가한다.                                                          |

### 4.12 GrowthTaskCompletion

| 항목      | 정의                                                                                 |
| --------- | ------------------------------------------------------------------------------------ |
| 객체명    | GrowthTaskCompletion                                                                 |
| 목적      | 사용자의 미션 완료 기록                                                              |
| 주요 필드 | `completion_id`, `user_id`, `growth_task_id`, `completed_at`, `earned_exp`, `status` |
| 관계      | User 1:N GrowthTaskCompletion                                                        |
| 정책      | 같은 사용자가 같은 일일 미션을 중복 완료할 수 없도록 제한한다.                       |

### 4.13 UserGrowthStat

| 항목      | 정의                                                                                                                             |
| --------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 객체명    | UserGrowthStat                                                                                                                   |
| 목적      | 사용자 레벨, 경험치, 활동 누적치 관리                                                                                            |
| 주요 필드 | `growth_stat_id`, `user_id`, `level`, `current_exp`, `total_exp`, `reading_score`, `news_score`, `english_score`, `health_score` |
| 관계      | User 1:1 UserGrowthStat                                                                                                          |
| 정책      | 경험치 누적에 따라 레벨을 자동 계산한다.                                                                                         |

### 4.14 CommunityPost

| 항목      | 정의                                                                                                                                                     |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 객체명    | CommunityPost                                                                                                                                            |
| 목적      | 전체/자유/레벨업 인증/취미 게시판 글 관리                                                                                                                |
| 주요 필드 | `post_id`, `user_id`, `board_type`, `title`, `body`, `is_anonymous`, `is_question`, `view_count`, `like_count`, `comment_count`, `share_count`, `status` |
| 관계      | User 1:N CommunityPost                                                                                                                                   |
| 정책      | 익명 글도 내부 `user_id`는 유지하되 화면에는 익명 표시한다.                                                                                              |

### 4.15 CommunityComment

| 항목      | 정의                                                                                      |
| --------- | ----------------------------------------------------------------------------------------- |
| 객체명    | CommunityComment                                                                          |
| 목적      | 게시글 댓글 및 대댓글 관리                                                                |
| 주요 필드 | `comment_id`, `post_id`, `user_id`, `parent_comment_id`, `body`, `is_anonymous`, `status` |
| 관계      | CommunityPost 1:N CommunityComment                                                        |
| 정책      | 삭제된 댓글은 본문을 마스킹하고 댓글 수 정합성을 유지한다.                                |

### 4.16 CommunityReaction

| 항목      | 정의                                                                                |
| --------- | ----------------------------------------------------------------------------------- |

| 관계      | User 1:N CommunityReaction                                                          |
| 정책      | 동일 사용자는 동일 대상에 동일 반응을 1회만 남길 수 있다.                           |

### 4.17 CommunityReport

| 항목      | 정의                                                                                                          |

| 관계      | User 1:N CommunityReport                                                                                      |
| 정책      | 신고 누적 기준에 따라 자동 숨김 또는 운영자 검토로 전환한다.                                                  |

| 정책      | 광고 로그는 분석용으로 분리 저장하며 개인정보 최소화를 적용한다.                          |

## 5. 객체 관계 요약

| 부모 객체             | 자식 객체            | 관계 | 설명                                               |
| --------------------- | -------------------- | ---- | -------------------------------------------------- |
| User                  | AuthIdentity         | 1:N  | 한 회원은 여러 로그인 수단을 가질 수 있다.         |
| User                  | PayrollPlan          | 1:N  | 한 회원은 월별 급여 계획을 여러 개 가진다.         |
| PayrollPlan           | FixedExpense         | 1:N  | 한 급여 계획에는 여러 고정지출이 연결된다.         |
| PayrollPlan           | SavingsPlan          | 1:N  | 한 급여 계획에는 여러 저축 계획이 연결된다.        |
| User                  | DailyBudget          | 1:N  | 한 회원은 날짜별 예산을 가진다.                    |
| DailyBudget           | VariableExpense      | 1:N  | 하루 예산에는 여러 변동지출이 연결된다.            |

## 7. 도메인별 데이터 흐름

### 7.1 급여 계획 생성 흐름

1. 사용자가 급여일과 수령 예정 급여를 입력한다.
2. 고정지출과 고정저축을 등록한다.
3. 일일 생활비 기준 금액을 설정한다.
4. 시스템은 월 예상 지출 금액과 예상 납치금액을 계산한다.
5. `PayrollPlan`, `FixedExpense`, `SavingsPlan`, `DailyBudget`에 저장한다.

### 7.2 일일 지출 기록 흐름

1. 사용자가 변동지출 금액과 항목을 입력한다.
2. `VariableExpense`가 생성된다.
3. 연결된 `DailyBudget.used_amount`가 증가한다.
4. `remaining_amount`, `over_amount`, `status`가 재계산된다.
5. 초과 시 `Notification`이 생성된다.

### 7.3 월말 납치금액 확정 흐름

1. 해당 월의 급여 계획을 조회한다.
2. 고정지출 완료금액, 변동지출 완료금액, 고정저축 완료금액을 합산한다.
3. 소비성 지출을 제외한 보호 금액을 확정한다.
4. 누적 납치금액 통계에 반영한다.
5. 목표 달성 시 알림과 이벤트 보상 후보를 생성한다.

## 8. 최종본 적용 규칙

본 데이터 모델은 급여납치 플랫폼의 모든 기능 개발, DB 설계, API 설계, QA 테스트, 운영 정책의 기준이다. 본 문서에 없는 신규 데이터 객체는 별도 추가가 아니라 기존 도메인의 확장 필드 또는 확장 테이블로 판단하며, 반드시 다음 기준을 통과해야 한다.

- 삭제/보존 정책이 연결되는가?
- 기존 계산식과 충돌하지 않는가?
- 광고/제휴 데이터와 재무 원천 데이터가 분리되는가?
