# 커뮤니티 기능 명세서 최종본

## 1. 문서 목적

본 문서는 전체 게시판, 자유 게시판, 레벨업 인증, 취미 게시판의 게시글 목록, 상세, 댓글, 좋아요, 조회수, 공유, 익명, 질문글, 신고 기능을 최종 확정한다. 커뮤니티는 직장인과 사용자 간 정보 공유·인증·소통을 담당한다.

## 공통 전제

| 구분        | 최종 기준                                                                     |
| ----------- | ----------------------------------------------------------------------------- |
| 서비스 유형 | 급여관리·가계부·자기계발·커뮤니티 결합형 모바일 애플리케이션                  |
| 핵심 사용자 | 직장인, 사회초년생, 아르바이트생, 소비관리 필요자                             |
| 핵심 지표   | 수령금액, 지출금액, 납치금액, 누적 납치금액, 목표 달성률, 일일 남은금액, 레벨 |
| 기본 통화   | KRW, 원 단위 정수                                                             |
| 금액 표시   | 천 단위 콤마 + `원` 표기                                                      |
| 금액 저장   | 숫자형 정수 저장, 화면 표시 시 포맷팅                                         |
| 음수 처리   | 납치금액은 0원 미만 표시 금지, 남은금액은 초과 상태 판단을 위해 음수 허용     |
| 인증 상태   | 비로그인 사용자는 로그인/회원가입/스플래시 외 핵심 기능 접근 불가             |
| 개인정보    | 급여·대출·저축·소비 내역은 민감 금융성 생활 데이터로 분류                     |
| 광고 데이터 | 급여·대출·저축 원천 데이터와 광고 추천/클릭 로그는 분리 저장                  |

## 공통 우선순위 정의

| 우선순위 | 의미                           | 출시 기준                            |
| -------- | ------------------------------ | ------------------------------------ |
| P1       | 핵심 급여관리 기능             | MVP 출시 전 반드시 완료              |
| P2       | 유지율·커뮤니티·운영 핵심 기능 | 정식 출시 전 완료 또는 단계적 활성화 |
| P3       | 수익화·고도화 기능             | 안정화 이후 활성화 가능              |

## 공통 오류 처리 원칙

1. 사용자가 수정 가능한 오류는 필드 하단 문구와 토스트로 안내한다.
2. 서버·네트워크 오류는 재시도 버튼을 제공한다.
3. 금액 계산 오류는 저장을 차단하고 원인 필드를 강조한다.
4. 중복 요청은 클라이언트 버튼 비활성화와 서버 멱등성 키로 방지한다.
5. 저장 성공 후 관련 화면의 수치와 목록은 즉시 재조회 또는 로컬 상태 갱신으로 동기화한다.

## 3. 입력/출력 필드

| 항목        | 필수 | 타입    | 설명                | 검증       |
| ----------- | ---- | ------- | ------------------- | ---------- |
| boardType   | Y    | enum    | 게시판 유형         | 허용값     |
| title       | Y    | string  | 제목                | 1자 이상   |
| body        | Y    | string  | 본문                | 1자 이상   |
| isAnonymous | Y    | boolean | 익명 여부           | true/false |
| isQuestion  | Y    | boolean | 질문 여부           | true/false |
| status      | Y    | enum    | 정상/숨김/삭제/검토 | 허용값     |

## 4. 사용자 및 시스템 흐름

### 1. 목록 조회

사용자는 게시판 탭을 선택해 게시글 목록을 최신순 또는 인기순으로 조회한다.

### 2. 상세 조회

게시글을 선택하면 상세, 댓글, 좋아요, 조회수 정보를 확인한다.

### 3. 상호작용

좋아요, 댓글, 공유, 신고 기능을 수행한다.

### 4. 신고 처리

신고가 접수되면 운영자 검토 큐에 반영되며 기준 초과 시 자동 숨김 가능하다.

## 5. 검증 규칙

| 검증 항목 | 기준           | 실패 처리                                       |
| --------- | -------------- | ----------------------------------------------- |
| 게시판    | 필수           | 게시판을 선택해주세요.                          |
| 제목/본문 | 필수           | 제목과 본문을 입력해주세요.                     |
| 금칙어    | 운영 정책 적용 | 운영 정책에 맞지 않는 표현이 포함되어 있습니다. |
| 신고      | 사유 필수      | 신고 사유를 선택해주세요.                       |

## 6. 예외 처리

| 예외 상황   | 처리 방식                     | 사용자 안내                           |
| ----------- | ----------------------------- | ------------------------------------- |
| 게시글 없음 | 빈 상태 표시                  | 아직 게시글이 없습니다.               |
| 신고 누적   | 숨김 또는 검토 상태           | 운영 정책에 따라 숨김 처리되었습니다. |
| 익명 글     | 일반 사용자에게 작성자 비노출 |                                       |

## 7. 데이터 객체

| 데이터 객체   | 주요 필드                                                           | 처리 기준   |
| ------------- | ------------------------------------------------------------------- | ----------- |
| CommunityPost | id, userId, boardType, title, body, isAnonymous, isQuestion, status | 게시글 원장 |
| Comment       | id, postId, userId, body, status                                    | 댓글        |
| PostReaction  | postId, userId, type                                                | 좋아요/공유 |
| Report        | targetType, targetId, userId, reason, status                        | 신고        |

## 8. 수락 기준

| 수락 기준 | 완료 판정          | 테스트 조건            |
| --------- | ------------------ | ---------------------- |
| 목록 표시 | 게시판별 목록 노출 | 탭 전환 테스트         |
| 좋아요    | 중복 좋아요 차단   | 같은 사용자 2회 클릭   |
| 신고      | 신고 이력 저장     | 신고 후 운영자 큐 확인 |

## 10. 보안·개인정보 기준

1. 본 기능에서 생성되는 사용자 개인 데이터는 로그인 사용자 본인만 조회·수정·삭제할 수 있다.
2. 급여, 지출, 저축, 소비, 프로필, 알림 수신 설정은 사용자 식별 데이터와 분리 가능한 구조로 관리한다.
3. 관리자 또는 운영자 접근이 필요한 경우 역할 기반 권한을 적용하고 모든 조작을 감사 로그로 저장한다.
4. 광고·제휴 분석에 민감 급여 데이터를 직접 사용하지 않는다.

## 11. 최종 완료 판정

본 문서는 커뮤니티 기능 명세서의 구현 범위, 입력값, 출력값, 검증, 예외, 데이터, 수락 기준을 모두 확정했다. 본 문서 기준으로 추가 기획 없이 구현·테스트·검수가 가능하므로 문서상·이론상 추가 작업은 필요하지 않다.
