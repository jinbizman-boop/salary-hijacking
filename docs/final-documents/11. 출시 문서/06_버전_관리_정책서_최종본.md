# 06. 급여납치 버전 관리 정책서 최종본

> 문서 상태: 최종본  
> 적용 범위: 급여납치 모바일 애플리케이션 iOS/Android 정식 출시  
> 작성 기준일: 2026-06-15  
> 기준 시간대: Asia/Seoul  
> 문서 원칙: 본 문서는 문서상·이론상 추가 기획 작업이 필요 없도록 출시 준비, 검수, 등록, 마케팅, 버전, 롤백 기준을 완결형으로 정의한다.

---

## 1. 문서 목적

본 문서는 급여납치 앱, API 서버, DB 마이그레이션, 문서, 스토어 릴리즈의 버전 번호, 브랜치, 태그, 배포 기준, hotfix 기준을 통일하기 위한 최종 정책서다.

## 2. 버전 체계

급여납치는 Semantic Versioning 기반의 `MAJOR.MINOR.PATCH` 체계를 사용한다.

| 구분   | 형식           | 증가 기준                                                    | 예시            |
| ------ | -------------- | ------------------------------------------------------------ | --------------- |
| MAJOR  | X.0.0          | 호환성 깨지는 대규모 변경, 핵심 IA 변경, DB 구조 대규모 변경 | v2.0.0          |
| MINOR  | 1.X.0          | 신규 기능 추가, 기존 기능 하위 호환 유지                     | v1.1.0          |
| PATCH  | 1.0.X          | 버그 수정, 보안 패치, 문구 수정, 작은 개선                   | v1.0.1          |
| HOTFIX | 1.0.X-hotfix.N | 운영 중 긴급 수정 브랜치/태그                                | v1.0.1-hotfix.1 |
| RC     | 1.0.0-rc.N     | 출시 후보 빌드                                               | v1.0.0-rc.1     |
| BETA   | 1.0.0-beta.N   | 외부/내부 테스트 빌드                                        | v1.0.0-beta.1   |

## 3. 앱 버전/빌드 번호 기준

| 플랫폼  | 사용자 노출 버전           | 내부 빌드 번호  | 예시                      |
| ------- | -------------------------- | --------------- | ------------------------- |
| iOS     | CFBundleShortVersionString | CFBundleVersion | 1.0.0 / 100               |
| Android | versionName                | versionCode     | 1.0.0 / 100000            |
| API     | API release tag            | git commit SHA  | api-v1.0.0                |
| DB      | migration version          | timestamp       | 202606150001_create_users |
| 문서    | document version           | 최종본 일자     | DOC-v1.0.0-20260615       |

## 4. 버전 증가 규칙

| 변경 유형                    | 버전 증가                    | 예시                                   |
| ---------------------------- | ---------------------------- | -------------------------------------- |
| 앱 첫 정식 출시              | v1.0.0                       | 급여관리 핵심 기능 출시                |
| 지출 기록 오류 수정          | PATCH                        | v1.0.1                                 |
| 알림 설정 기능 추가          | MINOR                        | v1.1.0                                 |
| DB 스키마 하위 호환 변경     | MINOR                        | v1.2.0                                 |
| API 응답 구조 변경           | MAJOR                        | v2.0.0                                 |
| 로그인 보안 취약점 긴급 수정 | PATCH/HOTFIX                 | v1.0.2                                 |
| 스토어 문구 오탈자 수정만    | PATCH 또는 metadata revision | v1.0.1 또는 스토어 메타데이터 업데이트 |

## 5. 브랜치 전략

| 브랜치           | 목적             | 보호 규칙                               |
| ---------------- | ---------------- | --------------------------------------- |
| `main`           | 정식 배포 코드   | 직접 push 금지, PR 승인 필수            |
| `develop`        | 다음 배포 통합   | CI 통과 필수                            |
| `feature/*`      | 기능 개발        | develop에서 분기, PR 병합               |
| `fix/*`          | 일반 버그 수정   | develop에서 분기                        |
| `release/vX.Y.Z` | 출시 후보 안정화 | QA/보안/정책 검수                       |
| `hotfix/vX.Y.Z`  | 운영 긴급 수정   | main에서 분기 후 main/develop 동시 반영 |
| `docs/*`         | 문서 변경        | 관련 담당 검수 후 병합                  |

## 6. 태그 정책

| 태그                  | 의미                 | 예시                |
| --------------------- | -------------------- | ------------------- |
| `app-v1.0.0`          | 앱 정식 릴리즈       | app-v1.0.0          |
| `api-v1.0.0`          | API 정식 릴리즈      | api-v1.0.0          |
| `db-v1.0.0`           | DB 마이그레이션 기준 | db-v1.0.0           |
| `docs-v1.0.0`         | 문서 세트 기준       | docs-v1.0.0         |
| `app-v1.0.1-hotfix.1` | 긴급 패치            | app-v1.0.1-hotfix.1 |

## 7. 출시 후보 기준

`release/vX.Y.Z` 브랜치는 다음 조건이 모두 충족될 때 생성한다.

1. 기획 범위 확정.
2. P1 기능 개발 완료.
3. DB 마이그레이션 파일 작성 완료.
4. API 명세와 실제 구현 일치.
5. QA 테스트 케이스 초안 완료.
6. 스토어 등록 문구 초안 완료.
7. 개인정보/약관/권한 정책 검수 완료.

| 항목   | 기준                               |
| ------ | ---------------------------------- |
| 기능   | P1 기능 100%, P2 승인 범위 통과    |
| 테스트 | S0/S1/S2 버그 0건                  |
| 보안   | 고위험 취약점 0건                  |
| 성능   | 핵심 API p95 기준 충족             |

| 스토어 | 메타데이터/스크린샷/정책 등록 완료 |
| 운영   | 모니터링/CS/장애 대응 준비 완료    |

## 9. Hotfix 정책

### 9.1 Hotfix 대상

다음 이슈는 hotfix 대상이다.

- 앱 실행 불가.
- 로그인 불가.
- 급여/지출/납치금액 계산 오류.
- 타 사용자 데이터 노출.
- 결제/광고/알림으로 인한 심각한 사용자 피해.
- 스토어 심사 거절 원인 중 즉시 수정 가능한 항목.
- 법적/개인정보 리스크가 있는 문구 또는 데이터 처리 오류.

### 9.2 Hotfix 절차

1. Release Manager가 hotfix 필요 여부 판정.
2. `main`에서 `hotfix/vX.Y.Z` 브랜치 생성.
3. 최소 수정 원칙으로 패치.
4. 관련 테스트만 빠르게 수행하되 보안/계산/로그인은 생략 금지.
5. 태그 생성.
6. 스토어 긴급 심사 또는 단계적 배포.
7. 수정 내용을 `develop`에 역병합.
8. 릴리즈 노트 작성.

## 10. DB 마이그레이션 버전 정책

| 항목      | 기준                                           |
| --------- | ---------------------------------------------- |
| 파일명    | `YYYYMMDDHHMM_description.sql`                 |
| 적용 순서 | timestamp 오름차순                             |
| 롤백      | 데이터 손실 가능 마이그레이션은 사전 백업 필수 |
| 금지      | 운영 DB 직접 수동 변경                         |
| 승인      | Tech Lead + DBA/Backend Owner 승인             |
| 검증      | staging DB 적용 후 production 적용             |

## 11. API 버전 정책

| 항목                | 기준                               |
| ------------------- | ---------------------------------- |
| 기본 경로           | `/api/v1`                          |
| 하위 호환           | v1 내에서는 기존 필드 삭제 금지    |
| 필드 추가           | 선택 필드로 추가 가능              |
| 필드 삭제/의미 변경 | v2 이상에서만 허용                 |
| deprecation         | 최소 1개 minor 버전 이상 사전 공지 |

## 12. 문서 버전 정책

| 문서 유형      | 버전 증가 기준                         |
| -------------- | -------------------------------------- |
| 기획 문서      | 기능 범위 변경 시 MINOR 이상           |
| API 문서       | endpoint/응답 변경 시 PATCH 또는 MINOR |
| 보안/정책 문서 | 법/스토어 정책 대응 시 즉시 PATCH      |
| 출시 문서      | 스토어 등록 정보/버전 변경 시 PATCH    |

## 13. 버전 관리 완료 기준

버전 관리는 다음 조건이 충족되면 완료로 본다.

- 앱, API, DB, 문서 버전이 추적 가능하다.
- 브랜치와 태그가 배포 이력과 일치한다.
- hotfix가 main/develop 양쪽에 반영된다.
- 릴리즈 노트와 스토어 버전 정보가 일치한다.
- DB 마이그레이션과 롤백 기록이 남아 있다.
