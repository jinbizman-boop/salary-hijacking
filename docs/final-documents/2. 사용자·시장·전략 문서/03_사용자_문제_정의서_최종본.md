# 사용자 문제 정의서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼이 해결해야 할 사용자 문제를 최종 정의한다. 제품 기능은 사용자의 실제 문제를 해결하기 위한 수단이며, 본 문서의 문제 정의를 벗어난 기능은 초기 범위에서 제외하거나 후순위로 관리한다.

## 2. 핵심 문제 한 문장 정의

급여 생활자는 월급을 받는 순간부터 고정지출, 변동지출, 충동소비, 자기계발 실패가 뒤섞여 **이번 급여에서 실제로 지켜낸 돈과 오늘 써도 되는 돈을 명확히 알지 못한다.**

급여납치는 이 문제를 **급여일 중심 계획, 일일 예산, 납치금액 시각화, 자기계발 루틴, 커뮤니티 인증**으로 해결한다.

## 4. 주요 문제 상세 정의

### 4.1 문제 1: 월급 소진 문제

#### 문제 정의

사용자는 월급을 받았을 때 계좌 잔액이 커진 것을 보고 여유가 생겼다고 느끼지만, 실제로는 고정지출과 예정 소비가 이미 상당 부분을 차지하고 있다. 그러나 사용자는 “실제로 지켜야 할 돈”과 “써도 되는 돈”을 구분하지 못해 월급이 빠르게 소진된다.

#### 사용자 증상

- 월급 직후 소비가 증가한다.
- 월말에 잔액 부족을 경험한다.
- 카드값이 예상보다 크게 나온다.
- 매달 같은 문제가 반복된다.

#### 원인

1. 급여 입금액과 가용 생활비를 구분하지 않는다.
2. 고정지출을 급여 수령 전에 미리 차감하지 않는다.
3. 지출 통제를 월말에 시작한다.
4. 절약의 성과를 즉시 볼 수 없다.

#### 제품 해결

- 급여 수령금액, 지출금액, 납치금액을 홈 상단에 고정 표시한다.
- 지출 예정금액을 급여 수령 즉시 차감해 예상 납치금액을 산출한다.
- 누적 납치금액을 성과 지표로 제공한다.

#### 성공 기준

- 사용자가 첫 사용 3분 이내에 이번 급여의 예상 납치금액을 확인한다.
- 사용자가 급여일 이후 7일 내 최소 3회 이상 일일 예산을 확인한다.

### 4.2 문제 2: 고정지출 누락 문제

#### 문제 정의

사용자는 구독료, 대출상환, 보험료, 통신비, 카드값 등 반복적으로 빠져나가는 돈을 기억에 의존한다. 이로 인해 급여 수령 직후 실제 가용금액을 과대평가하고, 결제일마다 계획이 무너진다.

#### 사용자 증상

- 자동결제일에 예상치 못한 출금이 발생한다.
- 구독료가 여러 개 누적되어도 총액을 모른다.
- 대출상환·보험료·통신비가 생활비를 압박한다.
- 결제 예정 알림이 없어 지출 계획을 수정하지 못한다.

#### 원인

1. 고정지출 항목이 흩어져 있다.
2. 결제일, 항목, 금액, 상태가 한 화면에서 보이지 않는다.
3. 고정지출과 변동지출이 구분되지 않는다.
4. “납부완료” 상태 관리가 없다.

#### 제품 해결

- 고정지출 등록 테이블을 제공한다.
- 항목별 결제일, 분류, 금액, 납부상태를 관리한다.
- 급여 홈에서 금일 고정지출을 우선 노출한다.
- 결제 예정 알림과 납부완료 처리를 제공한다.

#### 성공 기준

- 온보딩 또는 계획 화면에서 고정지출 1개 이상 등록한 사용자 비율을 측정한다.
- 고정지출 알림 클릭 후 납부완료 처리율을 측정한다.

### 4.3 문제 3: 일일 예산 초과 문제

#### 문제 정의

사용자는 한 달 예산을 세워도 하루 단위 소비 기준이 없어 매일의 행동으로 연결하지 못한다. 전체 예산은 추상적이고, 소비 순간에 판단 가능한 기준이 없기 때문에 예산을 초과한다.

#### 사용자 증상

- 오늘 얼마나 썼는지 기억하지 못한다.
- 월초에 많이 쓰고 월말에 급격히 줄인다.
- 소액 지출을 과소평가한다.
- 예산 초과 여부를 늦게 안다.

#### 원인

1. 월 예산이 하루 행동 기준으로 변환되지 않는다.
2. 지출 기록이 번거롭다.
3. 사용금액과 남은금액이 실시간으로 표시되지 않는다.
4. 초과 알림이 행동을 바꾸는 순간에 제공되지 않는다.

#### 제품 해결

- 일일 설정금액, 사용금액, 남은금액을 홈에 고정 표시한다.
- 지출 추가 시 남은금액을 즉시 재계산한다.
- 초과 사용 시 색상과 알림으로 경고한다.
- 빠른 지출 입력을 제공한다.

#### 성공 기준

- 일일 지출 기록 사용자 중 7일 내 재기록률을 측정한다.
- 일일 예산 초과 후 다음 날 기록 재개율을 측정한다.

### 4.4 문제 4: 변동지출 기록 실패 문제

#### 문제 정의

가계부 기록은 사용자가 직접 반복 입력해야 하므로 피로도가 높다. 특히 커피, 편의점, 식사, 교통, 게임결제 등 소액 변동지출은 즉시 기록하지 않으면 누락된다.

#### 사용자 증상

- 앱 설치 후 며칠만 기록하고 중단한다.
- 지출 항목을 나중에 몰아서 입력하려다 포기한다.
- 카테고리 분류가 귀찮다.
- 실제 소비와 앱 기록이 달라진다.

#### 원인

1. 입력 단계가 많다.
2. 금액 입력 후 성과 변화가 바로 보이지 않는다.
3. 기록 보상이 약하다.
4. 반복 소비 템플릿이 없다.

#### 제품 해결

- 금액 중심 빠른 추가 입력을 제공한다.
- 입력 즉시 오늘 사용금액과 남은금액을 갱신한다.
- 자주 쓰는 항목을 추천한다.
- 기록 완료를 레벨업 경험치와 연결할 수 있다.

#### 성공 기준

- 지출 기록 평균 입력 시간을 10초 이하로 설계한다.
- 지출 기록 후 홈 재방문율을 측정한다.

### 4.5 문제 5: 자기계발 지속 실패 문제

#### 문제 정의

사용자는 돈을 아끼고 싶어 하지만 단순 절약만으로는 동기 유지가 어렵다. 절약한 돈이 자신의 성장으로 연결된다는 경험이 부족하면 앱 사용이 단순 기록으로 끝난다.

#### 사용자 증상

- 독서, 영어, 운동 목표를 자주 세우지만 지속하지 못한다.
- 자기계발 지출이 낭비인지 투자자인지 판단하기 어렵다.
- 금융관리 앱과 루틴 앱을 따로 써서 사용이 분산된다.
- 성장 기록이 누적되지 않는다.

#### 원인

1. 금융관리와 자기계발 루틴이 분리되어 있다.
2. 매일 실행 가능한 작은 미션이 없다.
3. 완료 보상이 없다.
4. 인증 공간이 없다.

#### 제품 해결

- LV UP 메뉴에서 독서, 뉴스, 영어, 건강 미션을 제공한다.
- 완료 시 경험치, 레벨, 활동 기록을 반영한다.
- 마이페이지에 자기관리 성과를 표시한다.
- 커뮤니티 인증 게시판과 연결한다.

#### 성공 기준

- 주간 LV UP 미션 완료율을 측정한다.
- LV UP 기능 사용자의 주간 재방문율을 측정한다.

### 4.6 문제 6: 커뮤니티 고립 문제

#### 문제 정의

급여, 소비, 절약, 대출, 저축, 자기계발은 개인적이고 민감한 주제다. 사용자는 익명성이 없으면 이야기하기 어렵고, 혼자 관리하면 쉽게 포기한다.

#### 사용자 증상

- 돈 관리 실패를 주변에 말하지 못한다.
- 현실적인 절약 경험을 얻기 어렵다.
- 자기계발 인증을 공유할 공간이 없다.
- 혼자 기록하면 앱을 떠난다.

#### 원인

1. 급여·소비 정보는 민감하다.
2. 일반 SNS는 비교와 과시가 강하다.
3. 금융 앱은 커뮤니티 기능이 약하다.
4. 커뮤니티 앱은 개인 급여관리 기능이 없다.

#### 제품 해결

- 익명 게시글과 질문글 옵션을 제공한다.
- 자유 게시판, 레벨업 인증, 취미 게시판을 분리한다.
- 좋아요, 댓글, 공유로 상호작용을 제공한다.
- 민감정보 노출 방지 정책을 적용한다.

#### 성공 기준

- 커뮤니티 첫 게시글 작성률을 측정한다.
- 인증 게시글 작성 후 7일 재방문율을 측정한다.

## 7. 기능 반영 원칙

1. 문제 해결과 직접 연결되지 않는 기능은 초기 범위에서 제외한다.
2. 사용자가 돈을 더 잘 쓰게 만드는 기능보다 먼저, 돈을 잃지 않게 만드는 기능을 우선한다.
3. 모든 금액 계산 기능은 사용자가 행동할 수 있는 숫자를 제공해야 한다.
4. 알림은 불안 유발이 아니라 행동 전환을 목적으로 한다.
5. 자기계발 기능은 절약의 보상과 재방문 동기를 만드는 역할을 한다.
6. 커뮤니티 기능은 비교 경쟁보다 인증과 회복을 중심으로 설계한다.

## 8. 최종 승인 기준

본 사용자 문제 정의서는 다음 조건을 충족하므로 최종본으로 확정한다.

- 월급 소진, 고정지출 누락, 예산 초과, 자기계발 지속 실패 문제를 모두 포함했다.
- 각 문제의 원인, 증상, 제품 해결, 성공 기준을 정의했다.
- 기능 우선순위를 문제 영향도와 연결했다.
- 후속 PRD, 화면설계, KPI, QA 문서의 기준 문제로 사용할 수 있다.
