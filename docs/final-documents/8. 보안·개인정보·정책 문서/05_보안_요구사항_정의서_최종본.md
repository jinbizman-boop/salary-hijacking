# 05. 보안 요구사항 정의서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 해킹, 정보 유출, 권한 오남용, API 남용, 커뮤니티 악용, 광고 데이터 오용을 방지하기 위한 보안 요구사항을 정의한다. 본 문서는 모바일 앱, API 서버, 관리자, Neon DB, Cloudflare, 파일 저장소, 푸시 시스템에 공통 적용된다.

## 2. 보안 목표

| 목표            | 설명                                                                   |
| --------------- | ---------------------------------------------------------------------- |
| 기밀성          | 급여·대출·저축·소비 데이터가 권한 없는 사용자에게 노출되지 않게 한다.  |
| 무결성          | 납치금액, 지출, 레벨, 게시글, 광고 로그가 임의 조작되지 않게 한다.     |
| 가용성          | 급여일/월말 집중 접속에도 핵심 서비스가 유지되게 한다.                 |
| 책임추적성      | 로그인, 관리자 조치, 신고 처리, 데이터 변경 이력을 감사 가능하게 한다. |
| 개인정보 최소화 | 목적에 필요한 데이터만 수집·저장·노출한다.                             |

## 3. 인증 보안 요구사항

| ID           | 요구사항                | 기준                                          | 우선순위 |
| ------------ | ----------------------- | --------------------------------------------- | -------- |
| SEC-AUTH-001 | 비밀번호 평문 저장 금지 | Argon2id 또는 bcrypt cost 12 이상 일방향 해시 | P1       |
| SEC-AUTH-002 | access token 짧은 만료  | 15분 이하                                     | P1       |
| SEC-AUTH-003 | refresh token rotation  | 재사용 감지 시 전체 세션 폐기                 | P1       |
| SEC-AUTH-004 | 소셜 로그인 PKCE 적용   | Authorization Code Flow + PKCE                | P1       |
| SEC-AUTH-005 | 기기별 세션 관리        | 사용자 설정에서 기기 로그아웃 가능            | P1       |
| SEC-AUTH-006 | 관리자 MFA 필수         | TOTP/WebAuthn 중 1개 이상                     | P1       |
| SEC-AUTH-007 | 반복 로그인 실패 제한   | 계정+IP 기준 rate limit                       | P1       |

## 4. 인가 보안 요구사항

| ID            | 요구사항                     | 기준                                                  |
| ------------- | ---------------------------- | ----------------------------------------------------- |
| SEC-AUTHZ-001 | 객체 소유권 검증             | 모든 user_id 기반 리소스는 토큰 subject와 소유권 확인 |
| SEC-AUTHZ-002 | 관리자 권한 분리             | CS, 운영, 보안, 슈퍼관리자 역할 분리                  |
| SEC-AUTHZ-003 | P1 데이터 관리자 기본 비노출 | 관리자 화면에서 급여/대출/저축 원문 금액 마스킹       |
| SEC-AUTHZ-004 | 익명글 내부 식별자 보호      | 신고 처리 권한자만 내부 작성자 ID 접근                |
| SEC-AUTHZ-005 | API 단위 권한 체크           | 라우트/서비스/DB 접근 계층에서 중복 검증              |

## 5. 암호화 요구사항

| 영역     | 기준                                                                |
| -------- | ------------------------------------------------------------------- |
| 전송구간 | HTTPS/TLS 1.2 이상, 권장 TLS 1.3, HSTS 적용                         |
| DB 저장  | Neon DB 저장 암호화 전제, P1 주요 필드는 앱 레벨 AES-256-GCM 암호화 |
| 비밀번호 | 복호화 불가 해시, salt 필수                                         |
| 토큰     | refresh token은 해시 저장, access token은 서버 로그 저장 금지       |
| 파일     | 비공개 업로드는 signed URL, 원본 파일 URL 직접 노출 금지            |
| 백업     | 백업 저장소 암호화, 접근권한 최소화                                 |
| 로그     | 이메일/금액/IP/토큰 마스킹 또는 해시 처리                           |

## 6. API 보안 요구사항

| ID          | 요구사항             | 기준                                                      |
| ----------- | -------------------- | --------------------------------------------------------- |
| SEC-API-001 | 입력값 검증          | DTO/schema validation 필수                                |
| SEC-API-002 | SQL Injection 방지   | ORM parameter binding 또는 prepared statement             |
| SEC-API-003 | Mass Assignment 방지 | 허용 필드 allowlist 적용                                  |
| SEC-API-004 | Rate Limit           | 로그인, 글쓰기, 댓글, 신고, 파일 업로드, 검색에 제한 적용 |
| SEC-API-005 | Pagination 제한      | limit 최대 100, 기본 20                                   |
| SEC-API-006 | CORS 제한            | 허용 도메인/앱 origin만 허용                              |
| SEC-API-007 | Idempotency          | 결제성/중복 생성 위험 작업은 Idempotency-Key 적용         |
| SEC-API-008 | 오류 응답 통일       | 내부 스택, SQL, secret, token 노출 금지                   |

## 7. Rate Limit 기준

| API             | 제한 기준                          | 초과 시 응답          |
| --------------- | ---------------------------------- | --------------------- |
| 로그인          | 계정+IP 기준 5회/10분              | 429 TOO_MANY_REQUESTS |
| 회원가입        | IP 기준 10회/1시간                 | 429                   |
| 비밀번호 재설정 | 이메일 기준 3회/1시간              | 429                   |
| 지출 추가       | 사용자 기준 120회/분               | 429                   |
| 게시글 작성     | 사용자 기준 10회/시간              | 429                   |
| 댓글 작성       | 사용자 기준 60회/시간              | 429                   |
| 신고            | 사용자 기준 30회/일                | 429                   |
| 파일 업로드     | 사용자 기준 30개/시간, 파일당 10MB | 413/429               |
| 검색            | 사용자 기준 120회/분               | 429                   |

## 8. 입력값 검증 기준

| 입력값       | 검증 기준                                         |
| ------------ | ------------------------------------------------- |
| 금액         | 정수, 0 이상, BIGINT 범위, 원 단위, 음수 금지     |
| 제목         | 1~100자, HTML 태그 escape                         |
| 본문         | 1~5000자, XSS 필터링, 금칙어 검사                 |
| 닉네임       | 2~20자, 특수문자 제한, 운영자/관리자 사칭 금지    |
| 첨부파일     | 허용 MIME/확장자만, 바이러스/악성 스캔, EXIF 제거 |
| OAuth state  | 세션 저장값과 일치 확인                           |
| redirect URL | allowlist 매칭                                    |

## 9. 모바일 앱 보안 요구사항

| 영역           | 기준                                                                                   |
| -------------- | -------------------------------------------------------------------------------------- |
| 안전 저장소    | 토큰은 Android Keystore/iOS Keychain 저장                                              |
| 루팅/탈옥 탐지 | 고위험 환경에서 재무 화면 보호 강화 경고                                               |
| 화면 캡처 보호 | 급여/지출 화면은 Android FLAG_SECURE 선택 적용 가능, iOS는 민감 화면 전환 시 블러 처리 |
| 앱 잠금        | 사용자가 설정한 경우 생체 인증/PIN 적용                                                |
| 딥링크         | 서명 검증 및 허용 경로만 처리                                                          |
| 네트워크       | 인증서 검증 실패 시 통신 차단                                                          |

## 10. 관리자 보안 요구사항

| 항목            | 기준                                               |
| --------------- | -------------------------------------------------- |
| 로그인          | 관리자 MFA 필수                                    |
| 접근망          | IP allowlist 또는 Zero Trust Access 적용           |
| 권한            | RBAC, 최소 권한, 직무별 메뉴 분리                  |
| 감사로그        | 로그인, 조회, 수정, 삭제, 제재, 다운로드 기록 저장 |
| 데이터 다운로드 | 슈퍼관리자 승인, 사유 입력, 다운로드 워터마크      |
| 세션            | 30분 미사용 시 자동 로그아웃                       |

## 11. 보안 이벤트 처리

| 이벤트                | 감지 기준                        | 대응                        |
| --------------------- | -------------------------------- | --------------------------- |
| refresh token 재사용  | 이미 폐기된 refresh token 재사용 | 사용자 전체 세션 폐기, 알림 |
| 대량 로그인 실패      | 동일 IP/계정 반복 실패           | IP 차단, CAPTCHA/지연       |
| 타인 데이터 접근 시도 | user_id mismatch                 | 403, 보안 로그              |
| 게시글 스팸           | 짧은 시간 반복 글쓰기            | 제한, 숨김, 운영 검토       |
| 파일 악성 탐지        | MIME 불일치/악성 시그니처        | 업로드 차단, 계정 제한      |
| 관리자 이상 행위      | 대량 조회/다운로드               | 보안 관리자 알림, 계정 정지 |

## 12. 최종 완료 선언

본 보안 요구사항 정의서는 인증, 인가, 암호화, API 보안, rate limit, 입력값 검증, 모바일 앱, 관리자, 보안 이벤트 대응 기준을 모두 포함한다. 따라서 본 문서는 문서상·이론상 더 이상 수행할 작업이 없는 최종본으로 확정한다.
