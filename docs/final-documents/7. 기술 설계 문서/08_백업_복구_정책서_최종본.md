# 08. 백업/복구 정책서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼의 데이터 손실 방지, DB 백업 주기, 파일 백업, 복구 절차, 장애 대응, 복구 검증 기준을 최종 확정한다.

## 2. 복구 목표

| 항목          | 최종 기준                                              |
| ------------- | ------------------------------------------------------ |
| RPO           | 핵심 DB 15분 이내, 파일 24시간 이내                    |
| RTO           | P1 장애 2시간 이내 핵심 기능 복구                      |
| 백업 대상     | Neon DB, R2 파일, 환경설정, migration, 관리자 감사로그 |
| 복구 우선순위 | 인증/급여/지출/예산 > 알림 > 커뮤니티 > 광고/통계      |
| 복구 원칙     | 데이터 무결성 우선, 불완전 복구 시 읽기 전용 모드 적용 |

## 3. 백업 대상

| 대상                | 백업 방식                                  |                     주기 |     보존 |
| ------------------- | ------------------------------------------ | -----------------------: | -------: |
| Neon DB prod        | Neon PITR/자동 백업 + 정기 logical export  | PITR 지속 + daily export |  30~90일 |
| DB schema/migration | GitHub repository                          |              commit 단위 |     영구 |
| R2 사용자 파일      | R2 versioning/lifecycle + 중요 파일 export |                    daily | 30~180일 |
| R2 배너/공지 파일   | R2 versioning                              |                  변경 시 |    180일 |
| 환경변수 목록       | secret 이름 목록만 문서화                  |                  변경 시 |     영구 |
| 관리자 감사로그     | DB + export                                |                    daily |      5년 |
| 배치 실행 로그      | log storage                                |                    daily |    180일 |

## 5. 복구 시나리오

### 5.1 DB 논리 오류

| 상황                 | 예시                 | 절차                                                                    |
| -------------------- | -------------------- | ----------------------------------------------------------------------- |
| 잘못된 migration     | 컬럼 삭제/제약 오류  | 배포 중단 → 영향 범위 확인 → backup branch 복원 → forward fix migration |
| 잘못된 batch         | 월말 납치금액 오계산 | affected rows 식별 → 이전 snapshot 기준 재계산 → 감사 로그 기록         |
| 사용자 데이터 오삭제 | soft delete 오작동   | deleted_at 복구 가능 여부 확인 → restore script 실행                    |

### 5.2 DB 장애

1. Cloudflare/API error rate 확인.
2. Neon status와 connection pool 상태 확인.
3. 쓰기 API를 제한하고 읽기 cache/degraded mode 적용.
4. 복구 가능하면 Neon branch/restore 기능으로 지정 시점 복원.
5. 복원 DB에서 무결성 검사를 수행한다.
6. production connection 전환 전 smoke test를 수행한다.
7. 운영 공지와 사후 보고서를 작성한다.

### 5.3 파일 손실

1. R2 object key와 file metadata를 비교한다.
2. version/lifecycle backup에서 복구한다.
3. 복구 불가한 첨부는 DB 상태를 `LOST`로 표시하고 사용자에게 재업로드 안내한다.
4. 프로필/배너 기본 이미지 fallback을 적용한다.

## 6. 복구 검증 체크리스트

| 검증          | 기준                              |
| ------------- | --------------------------------- |
| 사용자 로그인 | 정상 로그인 가능                  |
| 급여 홈       | 최신 급여계획/지출/예산 조회 가능 |
| 금액 계산     | 예상/누적 납치금액 재계산 일치    |
| 커뮤니티      | 게시글/댓글 목록 조회 가능        |
| 파일          | 프로필/첨부/배너 접근 가능        |
| 알림          | 읽음/미읽음 상태 유지             |
| 관리자        | 감사로그와 제재 상태 유지         |
| 배치          | 중복 실행 방지, last_run_at 확인  |

## 7. 백업 무결성 검증

| 주기 | 작업                                 |
| ---- | ------------------------------------ |
| 매일 | backup job 성공 여부 확인            |
| 매주 | staging restore rehearsal            |
| 매월 | production snapshot 기반 복구 리허설 |
| 분기 | 장애 대응 모의훈련, RPO/RTO 검증     |

## 8. 롤백과 복구 구분

| 구분          | 적용 상황             | 방식                                  |
| ------------- | --------------------- | ------------------------------------- |
| 코드 롤백     | 배포된 API/앱 오류    | 이전 Worker version 또는 앱 hotfix    |
| DB rollback   | 원칙적으로 금지       | forward-compatible migration으로 수정 |
| PITR 복구     | DB 손상/대규모 오삭제 | 지정 시점 복원 후 connection 전환     |
| 데이터 재계산 | 파생 데이터 오류      | 원천 데이터 기준 summary 재생성       |

## 9. 장애 등급별 대응

| 등급 | 예시                         | 대응 목표                       |
| ---- | ---------------------------- | ------------------------------- |
| P1   | 로그인/급여/지출 전체 장애   | 즉시 대응, 2시간 이내 핵심 복구 |
| P2   | 커뮤니티/알림/파일 일부 장애 | 4시간 이내 복구                 |
| P3   | 광고/통계/관리자 일부 장애   | 1영업일 이내 복구               |
| P4   | 문구/비핵심 UI 문제          | 정기 배포 처리                  |
