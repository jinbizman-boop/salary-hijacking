# 05. 확장성 설계서 최종본

## 1. 문서 목적

본 문서는 급여납치 플랫폼이 평균 5,000명 동시 접속과 최대 1,000만명 규모의 수시 접속/푸시/이벤트 트래픽을 문서상·이론상 감당할 수 있도록 확장성 기준을 최종 확정한다.

## 2. 확장성 목표

| 구분              | 목표                                             |
| ----------------- | ------------------------------------------------ |
| 평균 동시 접속    | 5,000명 안정 처리                                |
| 급여일 피크       | 일반일 대비 5~10배 트래픽 견딤                   |
| 푸시 후 순간 접속 | 알림 발송 직후 API burst 흡수                    |
| 커뮤니티 피크     | 인기글/인증글 조회 폭증 시 read cache 사용       |
| 광고/배너         | 핵심 기능 장애 없이 별도 degrade 가능            |
| DB 안정성         | connection 폭증 방지, pool/캐시/읽기 최적화 적용 |

## 3. 확장 전략 요약

| 계층     | 전략                                                      |
| -------- | --------------------------------------------------------- |
| 앱       | 로컬 cache, stale-while-revalidate, 중복 요청 제거        |
| Edge     | Cloudflare CDN/WAF/Rate Limit, Workers edge scale         |
| API      | 도메인별 route 분리, stateless worker, idempotency        |
| DB       | Neon pooled connection, 인덱스, 쿼리 timeout, 집계 테이블 |
| 파일     | R2 + CDN 캐싱, 원본 API 경유 금지                         |
| 푸시     | batch/queue 기반 분산 발송, 중복 발송 방지                |
| 커뮤니티 | 목록 cursor pagination, 인기글 cache, 신고/댓글 비동기    |

## 4. 트래픽 유형별 대응

| 트래픽        | 특징                | 대응                                                        |
| ------------- | ------------------- | ----------------------------------------------------------- |
| 급여 홈 조회  | 로그인 후 가장 빈번 | current summary endpoint, 30초 cache, DB aggregation 최소화 |
| 지출 추가     | 쓰기 집중           | 멱등성, transaction, 사용자별 rate limit                    |
| 계획 수정     | 월 1~수회           | 중복 월 계획 방지, DB unique index                          |
| 커뮤니티 목록 | read-heavy          | cursor pagination, 인기글 cache, CDN 가능 영역 분리         |
| 글쓰기/댓글   | write-heavy 가능    | rate limit, spam detection, 비동기 알림                     |
| 알림 읽음     | 다건 업데이트       | bulk endpoint, 비동기 처리                                  |
| 광고 노출     | 대량 로그           | write buffer, 집계 job, 핵심 DB transaction과 분리          |

## 5. DB 확장 기준

### 5.1 Connection Pooling

| 항목             | 기준                            |
| ---------------- | ------------------------------- |
| 기본 연결        | Neon pooled connection URL 사용 |
| Worker 요청      | 요청마다 긴 연결 유지 금지      |
| Long transaction | 금지, 3초 이하 지향             |
| Batch job        | 작업 단위 chunk 처리            |
| 연결 폭증        | retry backoff + rate limit 적용 |

### 5.2 인덱스 전략

| 테이블            | 필수 인덱스                                                  |
| ----------------- | ------------------------------------------------------------ |
| payroll_plans     | `(user_id, year_month DESC)`, active unique                  |
| daily_budgets     | `(user_id, budget_date DESC)`                                |
| variable_expenses | `(user_id, spent_at DESC)`, `(daily_budget_id)`              |
| fixed_expenses    | `(user_id, payroll_plan_id, expense_day)`                    |
| community_posts   | `(board_type, created_at DESC)`, `(status, created_at DESC)` |
| notifications     | `(user_id, created_at DESC)`, `(user_id, read_at)`           |
| ad_events         | `(campaign_id, occurred_at DESC)`                            |

### 5.3 집계/요약 테이블

| 요약                    | 목적                       | 갱신                  |
| ----------------------- | -------------------------- | --------------------- |
| user_money_summary      | 홈 누적 납치금액 빠른 조회 | 지출/월말 확정 시     |
| monthly_payroll_summary | 월별 급여/지출/납치금액    | 월말 batch            |
| community_post_stats    | 좋아요/댓글/조회수         | 이벤트 발생 시 upsert |
| growth_user_stats       | 레벨/경험치                | 미션 완료 시          |
| ad_daily_stats          | 광고 일별 집계             | batch                 |

## 6. API Cache 정책

| API                | cache 전략                              | TTL      |
| ------------------ | --------------------------------------- | -------- |
| 급여 홈 요약       | user-private memory/cache, client cache | 15~30초  |
| 레벨업 콘텐츠 목록 | public cache 가능                       | 10분     |
| 배너 목록          | public/segment cache                    | 5분      |
| 커뮤니티 인기글    | short cache                             | 30초~1분 |
| 알림 목록          | no shared cache, client cache           | 30초     |
| 관리자 통계        | server-side aggregate cache             | 1~5분    |

## 7. Queue/비동기 처리 기준

| 작업                | 처리 방식                                   |
| ------------------- | ------------------------------------------- |
| 푸시 발송           | notification 생성 후 dispatch job 처리      |
| 광고 이벤트 집계    | raw event 저장 후 batch rollup              |
| 월말 납치금액 확정  | scheduler chunk 처리                        |
| 대량 알림 생성      | 사용자 segment chunk 단위 insert            |
| 신고 후 운영자 알림 | 비동기 생성, 핵심 게시글 등록 blocking 금지 |

## 8. Degraded Mode

| 장애          | 대체 동작                                 |
| ------------- | ----------------------------------------- |
| DB 지연       | 최근 cache 표시, 쓰기 제한 안내           |
| 커뮤니티 부하 | 커뮤니티 목록만 일시 제한, 급여 기능 유지 |
| 광고 장애     | 광고 영역 숨김 또는 기본 배너 표시        |
| 푸시 장애     | 앱 내 알림은 저장, 외부 push 재시도       |
| 검색 장애     | 최신 목록 기본 조회로 fallback            |
| 관리자 장애   | 사용자 API와 분리, 서비스 지속            |

## 9. 확장 단계별 로드맵

| 단계    |    사용자 규모 | 확장 조치                                                         |
| ------- | -------------: | ----------------------------------------------------------------- |
| Stage 1 |   0~5만 가입자 | 단일 Worker/API, Neon pooled connection, 기본 인덱스              |
| Stage 2 |       5만~50만 | 집계 테이블, 광고 이벤트 batch, 커뮤니티 cache 강화               |
| Stage 3 |     50만~300만 | 도메인별 Worker 분리, read replica/branch 전략 검토, queue 고도화 |
| Stage 4 | 300만~1,000만+ | region 분산, read/write 분리, 전문 search/cache layer 추가        |

## 10. 병목 예방 체크리스트

| 항목          | 기준                                       |
| ------------- | ------------------------------------------ |
| DB connection | pooled URL 사용 여부 확인                  |
| 홈 API        | 단일 summary endpoint로 aggregation 최소화 |
| 목록 API      | offset 남용 금지, cursor 지원              |
| 파일          | API 서버가 파일 binary 중계 금지           |
| 푸시          | 사용자를 한 번에 전송하지 않고 chunk 처리  |
| 로그          | 동기 외부 전송으로 API 응답 지연 금지      |
| 배치          | 한 트랜잭션에 대량 사용자 처리 금지        |

## 11. 최종 수용 기준

| 기준        | 완료 조건                                             |
| ----------- | ----------------------------------------------------- |
| 대규모 접속 | 5,000명 동시 접속과 피크 트래픽 대비 전략이 있다.     |
| DB 안정성   | pooled connection, 인덱스, 집계, timeout 기준이 있다. |
| 서비스 분리 | 핵심 급여 기능이 커뮤니티/광고 장애와 분리된다.       |
| 확장 로드맵 | 사용자 규모별 확장 방식이 정의되어 있다.              |
| 운영 대응   | degrade mode와 병목 예방 기준이 있다.                 |
