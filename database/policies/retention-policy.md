# 데이터 보존/삭제 정책

문서 경로: `database/policies/retention-policy.md`
프로젝트: 급여납치 Salary Hijacking Platform
적용 범위: 사용자, 인증, 급여, 예산, 지출, 저축, 알림, LV UP, 커뮤니티, 첨부, 광고/제휴, 관리자 감사로그, 공지, 장애 대응, 백업, 로그
기준 DB: Neon PostgreSQL / PostgreSQL
기준 시간대: Asia/Seoul
기준 원칙: 개인정보 최소 보유, 서버 권위 삭제, 재무 원천 데이터 보호, 광고 데이터 분리, 감사 가능성 보장

---

## 1. 문서 목적

이 문서는 급여납치 플랫폼의 데이터 보존, 삭제, 익명화, 마스킹, 백업, 복구, 로그 보존 기준을 정의한다.

급여납치 플랫폼은 사용자의 급여, 고정지출, 고정저축, 일일예산, 변동지출, 납치금액, 커뮤니티 활동, 알림, 광고/제휴 이벤트, 관리자 운영 이력을 처리한다. 이 데이터 중 급여·지출·저축·예산·납치금액은 민감한 재무성 데이터로 분류하며, 광고/제휴 데이터와 직접 결합하거나 원문 로그에 기록하지 않는다.

본 정책은 다음 목표를 충족해야 한다.

1. 회원 탈퇴 시 사용자 식별성과 재무 원천 데이터를 안전하게 삭제 또는 비식별화한다.
2. 커뮤니티 익명글과 댓글은 서비스 맥락을 유지하되 작성자 식별 가능성을 제거한다.
3. 알림, 푸시 발송, 기기, 동의, 광고 이벤트, 감사로그의 보존 기간을 명확히 한다.
4. 백업 데이터는 복구 가능성과 삭제권 보장을 함께 만족하도록 만료 기준을 둔다.
5. 관리자 작업, 권한 변경, 장애 대응, 광고 승인 이력은 감사 가능성을 위해 장기 보존한다.
6. 모든 삭제·익명화·마스킹 작업은 서버 권위 API, 관리자 RBAC, 감사로그 기준으로 수행한다.

## 5. 회원 탈퇴 처리 정책

### 5.1 탈퇴 상태 전환

회원 탈퇴 요청이 접수되면 서버는 다음 순서로 처리한다.

1. 사용자 본인 인증 또는 관리자 대리 처리 권한을 검증한다.
2. `users.status`를 `WITHDRAWN`으로 변경한다.
3. `users.deleted_at`을 현재 시각으로 기록한다.
4. 활성 세션, refresh token, push token, device를 revoke한다.
5. 30일 유예 기간 동안 계정 복구 가능 상태를 유지할 수 있다.
6. 유예 기간 종료 후 식별 가능한 개인정보와 재무 원천 데이터를 삭제 또는 비식별화한다.

### 5.2 즉시 비활성화 대상

| 대상             | 처리                                                      |
| ---------------- | --------------------------------------------------------- |
| 로그인 인증      | `auth_identities.revoked_at` 기록                         |
| 기기             | `user_devices.status = REVOKED`, `revoked_at` 기록        |
| 푸시 알림        | 미발송 알림 `CANCELLED` 처리                              |
| 마케팅/광고 동의 | 철회 이력 추가                                            |
| 관리자 권한      | 사용자가 관리자라면 `admin_role_members.status = REVOKED` |

### 5.3 유예 기간 후 삭제 대상

다음 데이터는 탈퇴 유예 기간 종료 후 삭제한다.

- `fixed_expenses`
- `savings_plans`
- `daily_budgets`
- `variable_expenses`
- `payroll_calculation_snapshots`
- `payroll_plans`
- `user_settings`
- `user_growth_stats`
- `growth_task_completions`
- `community_reactions`
- 개인 첨부 파일 원본

### 5.4 익명화 대상

서비스 무결성, 신고 처리, 커뮤니티 맥락, 감사 필요성이 있는 데이터는 삭제 대신 익명화할 수 있다.

| 대상            | 익명화 정책                                                                           |
| --------------- | ------------------------------------------------------------------------------------- |
| 커뮤니티 게시글 | `user_id`는 시스템 익명 사용자 또는 삭제 사용자 참조로 전환, 표시명은 `탈퇴한 사용자` |
| 커뮤니티 댓글   | 작성자 표시 제거, 필요 시 본문 마스킹                                                 |
| 커뮤니티 신고   | 신고자 user_id 비식별화, 처리 결과는 보존                                             |
| 광고 이벤트     | `user_id = NULL`, session/device hash는 만료 후 삭제                                  |
| 관리자 감사로그 | actor가 탈퇴해도 감사 이벤트는 보존하되 개인정보 원문 없음                            |

## 6. 커뮤니티 익명글·삭제 정책

커뮤니티는 게시글 목록, 인기글, 댓글 흐름, 신고 처리의 정합성을 유지해야 하므로 무조건 물리 삭제하지 않는다.

### 6.1 사용자 직접 삭제

| 데이터 | 처리                                                                  |
| ------ | --------------------------------------------------------------------- |
| 게시글 | `status = DELETED`, `deleted_at` 기록, 본문을 삭제 안내 문구로 마스킹 |
| 댓글   | `status = DELETED`, `deleted_at` 기록, 본문을 삭제 안내 문구로 마스킹 |
| 첨부   | `attachments.status = DELETED`, object storage 파일 삭제 예약         |
| 반응   | 물리 삭제 가능                                                        |

### 6.2 운영자 삭제/숨김

| 조치      | 처리                                                      |
| --------- | --------------------------------------------------------- |
| 숨김      | `status = HIDDEN`, `moderation_reason` 기록               |
| 잠금      | 게시글 `status = LOCKED`, 댓글 추가 차단                  |
| 삭제      | `status = DELETED`, 본문 마스킹, 첨부 삭제                |
| 신고 처리 | `community_reports.status = RESOLVED/REJECTED/DUPLICATED` |

운영자 조치는 반드시 `admin_audit_logs`에 기록되어야 한다.

### 6.3 익명글

익명글은 화면에서 작성자 표시를 제거하는 UI 정책이지, DB 내부에서 무소유 데이터로 만드는 정책이 아니다. DB에는 abuse 대응과 본인 수정/삭제 권한 검증을 위해 `user_id`를 보존한다. 단, 탈퇴 후에는 작성자 식별 가능성을 제거한다.

## 7. 알림·푸시 데이터 보존 정책

### 7.1 알림 본문

`notifications`는 사용자에게 보여준 인앱 알림 이력이다. 기본 보존 기간은 180일이다.

| 상태        | 처리                         |
| ----------- | ---------------------------- |
| `SCHEDULED` | 발송 전 취소 가능            |
| `SENT`      | 180일 후 삭제                |
| `READ`      | 180일 후 삭제                |
| `FAILED`    | 장애 분석 후 180일 이내 삭제 |
| `CANCELLED` | 90일 후 삭제 가능            |
| `EXPIRED`   | 만료 후 30일 이내 삭제 가능  |

알림 `payload`에는 급여액, 지출액, 저축액, 납치금액, 이메일, 전화번호, 토큰 원문을 저장하지 않는다.

### 7.2 발송 이력

`notification_deliveries`는 provider 응답, 실패 사유, 재시도 횟수를 보존한다. 푸시 토큰 원문은 저장하지 않고 `user_devices.push_token_hash`만 참조한다.

## 8. 재무 데이터 보존/삭제 정책

급여납치의 재무 데이터는 사용자의 급여 생활과 소비 패턴을 직접적으로 드러낼 수 있으므로 가장 높은 보호 수준을 적용한다.

### 8.1 원문 보존 대상

활성 사용자에게 서비스 제공을 위해 다음 데이터는 원문 보존한다.

- 월별 급여계획
- 고정지출
- 고정저축
- 일일예산
- 변동지출
- 계산 스냅샷

### 8.2 원문 저장 금지 위치

다음 위치에는 재무 원천 데이터 원문 저장을 금지한다.

| 위치              | 금지 데이터                                             |
| ----------------- | ------------------------------------------------------- |
| 애플리케이션 로그 | 급여액, 지출액, 저축액, 납치금액, 계좌/카드/대출 정보   |
| 관리자 감사로그   | 변경 전후 재무 원문 전체                                |
| 광고 이벤트       | 급여/소비/저축/대출 기반 타겟팅 원천 데이터             |
| 커뮤니티 본문     | 타인의 금융 개인정보, 계좌/카드/주민등록/대출 계약 정보 |
| 알림 payload      | 급여/지출 상세 원문, 토큰, 개인정보                     |

### 8.3 통계 보존

사용자 단위 식별이 제거된 월별/일별 집계 통계는 서비스 품질 개선을 위해 보존할 수 있다. 단, 다음 조건을 충족해야 한다.

1. 개별 사용자를 재식별할 수 없어야 한다.
2. 최소 k-익명성 기준을 충족해야 한다.
3. 급여액·지출액·저축액 원문이 아닌 구간화 또는 집계값이어야 한다.
4. 광고 타겟팅에 직접 사용하지 않아야 한다.

## 10. 관리자 감사로그 보존 정책

`admin_audit_logs`는 다음 이벤트를 반드시 기록한다.

1. 관리자 역할 부여/해제
2. 사용자 제재/복구
3. 광고 파트너 생성/승인/중지
4. 광고 캠페인 승인/거절/긴급중지
5. 공지 생성/수정/삭제/발행
6. 커뮤니티 신고 처리
7. 첨부 격리/삭제
8. 장애 등록/완화/해결
9. 보안 정책 변경
10. 배포/롤백 관련 운영 조치

감사로그 보존 기간은 최소 5년이다. 보안 사고, 권한 오남용, 개인정보 사고, 정산 분쟁과 관련된 감사로그는 7년 보존한다.

감사로그에는 다음 원문을 저장하지 않는다.

- 이메일
- 전화번호
- IP 원문
- User-Agent 원문
- access token / refresh token
- push token
- database URL
- API key / secret key / private key
- 급여액, 지출 상세, 저축 상세, 납치금액 원문

## 11. 백업 보존/삭제 정책

### 11.1 백업 목적

백업은 장애 복구, 배포 롤백, 데이터 손상 대응을 위한 목적으로만 사용한다.

### 11.2 보존 기간

| 백업 유형             |               보존 기간 | 정책                   |
| --------------------- | ----------------------: | ---------------------- |
| 자동 PITR/WAL         | 인프라 플랜 기준 7~35일 | 만료 자동 삭제         |
| 일일 snapshot         |                  7~30일 | 운영 환경 기준 적용    |
| 배포 전 수동 snapshot |                     7일 | 배포 안정화 후 삭제    |
| 장애 대응 snapshot    |       사고 종료 후 30일 | 사후 분석 종료 후 삭제 |
| 법적 보존 snapshot    |        사안별 승인 기간 | 보안/법무 승인 필수    |

### 11.3 탈퇴 사용자와 백업

탈퇴 후 운영 DB에서 삭제·익명화가 완료되더라도 백업에는 최대 백업 보존 기간 동안 과거 데이터가 남을 수 있다. 백업은 개별 사용자 복원 목적으로 사용하지 않으며, 전체 장애 복구 시 복구 후 탈퇴/삭제 요청을 재적용해야 한다.

## 12. 로그 보존 정책

| 로그 유형              | 보존 기간 | 원칙                                         |
| ---------------------- | --------: | -------------------------------------------- |
| API access log         |   30~90일 | user_id, request_id 중심, 민감정보 원문 금지 |
| API error log          |      90일 | stack trace에 개인정보/재무 데이터 포함 금지 |
| Admin access log       |       1년 | 관리자 접근 추적                             |
| Auth security log      |       1년 | 로그인 실패, 비정상 접근 탐지                |
| WAF/CDN log            |  30~180일 | IP는 hash 또는 보안 저장소 기준              |
| CI/CD log              |     180일 | secret masking 필수                          |
| Payment/settlement log |       5년 | 광고 정산 집계 중심                          |

로그에는 다음 값이 포함되면 안 된다.

- `Authorization` header
- Cookie 원문
- refresh token
- push token
- database URL
- secret key
- private key
- 이메일/전화번호 원문
- 급여/지출/저축/납치금액 원문

## 13. 첨부 파일 보존/삭제 정책

첨부 파일은 DB와 object storage를 함께 삭제해야 한다.

| 상황                    | 처리                                                          |
| ----------------------- | ------------------------------------------------------------- |
| 사용자가 게시글 삭제    | 첨부 `status = DELETED`, object storage 삭제 예약             |
| 운영자가 정책 위반 격리 | `status = QUARANTINED`, 관리자만 접근                         |
| 신고 처리 후 삭제       | object storage 원본 삭제, checksum만 감사 목적 제한 보존 가능 |
| 사용자 탈퇴             | 개인 첨부 파일 30일 이내 삭제                                 |
| PDF 첨부                | 민감정보 포함 여부 스캔 후 정책 위반 시 격리                  |

object storage 삭제 작업은 idempotent해야 하며, 실패 시 재시도 큐에 기록한다.

---

## 14. 데이터 삭제 작업 순서

회원 탈퇴 또는 관리자 삭제 작업은 다음 순서를 따른다.

1. 요청자 인증 및 권한 확인
2. 삭제 대상 user_id 또는 target_id 잠금
3. 활성 세션 및 기기 revoke
4. 미발송 알림 취소
5. 재무 원천 데이터 삭제
6. LV UP 개인 완료 이력 삭제
7. 커뮤니티 반응 삭제
8. 커뮤니티 글/댓글 익명화 또는 마스킹
9. 첨부 파일 삭제 예약
10. 광고 이벤트 user_id 제거
11. 관리자 감사로그 기록
12. 백업 보존 기간 만료 대기
13. 삭제 검증 리포트 생성

삭제 작업은 중간 실패에 대비해 재실행 가능해야 한다.

---

## 15. 삭제 검증 기준

삭제/익명화 작업 완료 후 다음 검증을 수행한다.

| 검증 항목   | 기준                                                |
| ----------- | --------------------------------------------------- |
| 사용자 상태 | `users.status`가 `WITHDRAWN` 또는 `DELETED`         |
| 인증 식별자 | 활성 identity 없음                                  |
| 기기        | 활성 device 없음                                    |
| 재무 데이터 | 해당 user_id의 급여/지출/저축/예산 원천 데이터 없음 |
| 알림        | 미발송 알림 없음                                    |
| 커뮤니티    | 작성자 식별 가능 정보 제거                          |
| 광고 이벤트 | user_id 제거 또는 삭제                              |
| 첨부        | object storage 삭제 확인                            |
| 감사로그    | 삭제 요청과 처리 결과 기록                          |

## 16. 보존 연장 정책

다음 사유가 있는 경우 보존 기간을 연장할 수 있다.

1. 법적 분쟁
2. 보안 사고 조사

6. 장애 원인 분석

보존 연장은 관리자 RBAC 승인과 `admin_audit_logs` 기록이 필수다.

## 17. 운영 Runbook

### 17.1 정기 삭제 작업

| 주기   | 작업                                                |
| ------ | --------------------------------------------------- |
| 매일   | 만료 알림, 만료 광고 이벤트, 만료 첨부 삭제 큐 처리 |
| 매주   | 탈퇴 유예 종료 사용자 삭제/익명화                   |

| 분기별 | 백업 보존 정책 점검, 감사로그 샘플링 검토           |
| 반기별 | 개인정보 처리·광고 데이터 분리 정책 점검            |

### 17.2 장애 시 처리

삭제 작업 중 장애가 발생하면 다음 절차를 따른다.

1. 작업 중단 및 대상 row lock 해제 확인
2. 실패 지점과 request_id 기록
3. `operational_incidents` 등록
4. 재시도 가능성 확인
5. 데이터 부분 삭제 상태 검증
6. 관리자 감사로그 기록
7. 사용자 영향이 있으면 공지 또는 개별 안내

## 18. SQL 정책 예시

### 18.1 탈퇴 사용자 재무 데이터 삭제 예시

```sql
DELETE FROM public.variable_expenses WHERE user_id = :user_id;
DELETE FROM public.daily_budgets WHERE user_id = :user_id;
DELETE FROM public.fixed_expenses WHERE user_id = :user_id;
DELETE FROM public.savings_plans WHERE user_id = :user_id;
DELETE FROM public.payroll_calculation_snapshots WHERE user_id = :user_id;
DELETE FROM public.payroll_plans WHERE user_id = :user_id;
```

### 18.2 광고 이벤트 비식별화 예시

```sql
UPDATE public.ad_events
SET user_id = NULL
WHERE user_id = :user_id;
```

### 18.3 커뮤니티 게시글 익명화 예시

```sql
UPDATE public.community_posts
SET
  is_anonymous = true,
  updated_at = now()
WHERE user_id = :user_id
  AND status IN ('PUBLISHED', 'LOCKED', 'HIDDEN');
```

### 18.4 알림 만료 삭제 예시

```sql
DELETE FROM public.notification_deliveries
WHERE notification_id IN (
  SELECT notification_id
  FROM public.notifications
  WHERE created_at < now() - interval '180 days'
);

DELETE FROM public.notifications
WHERE created_at < now() - interval '180 days';
```

## 19. 금지 사항

다음 행위는 금지한다.

1. 사용자 탈퇴 후 급여·지출·저축 원천 데이터를 무기한 보존하는 행위
2. 광고 이벤트에 급여액, 지출액, 저축액, 납치금액을 저장하는 행위
3. 관리자 감사로그에 secret, token, 이메일, 전화번호, 원문 IP를 저장하는 행위
4. 커뮤니티 익명글의 작성자 정보를 API 응답으로 노출하는 행위
5. 삭제 요청을 처리하면서 감사로그를 남기지 않는 행위
6. 백업 데이터를 사용자 단위 복원 용도로 임의 사용하는 행위
7. object storage 파일만 삭제하고 DB 첨부 상태를 갱신하지 않는 행위
8. DB row만 삭제하고 object storage 파일을 방치하는 행위
9. 로그에 요청 본문 전체를 무차별 저장하는 행위
10. 운영자가 RBAC 승인 없이 보존 기간을 연장하는 행위

## 21. 최종 판정

`database/policies/retention-policy.md`는 급여납치 플랫폼의 회원 탈퇴, 백업, 커뮤니티 익명글, 알림, 로그, 재무 데이터, 광고/제휴, 관리자 감사로그, 첨부 파일, 장애 대응 데이터를 포괄하는 최종 데이터 보존/삭제 정책 문서다.

이 문서는 문서상·이론상 파일 단위 기준으로 더 이상 추가 작성이 필요 없는 최종본으로 사용한다. 단, 프로젝트 종합 완성도 100%는 실제 DB 마이그레이션, API 삭제 작업, 관리자 RBAC, object storage 삭제 큐, CI/CD 검증이 본 정책과 일치하게 구현되어야 최종 확정된다.
