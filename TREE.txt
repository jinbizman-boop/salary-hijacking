# salary-hijacking-platform TREE.txt

문서 상태: 최종본
파일 위치: TREE.txt
프로젝트: 급여납치 Salary Hijacking Platform
기준 구조: salary-hijacking-platform.zip
기준 원칙: 루트 계약 → 문서/DB → 공통 패키지 → 서버 → 앱 → 인프라/자동화 → 보안/QA/운영/출시
작성 목적: 프로젝트 전체 폴더·파일 구조, 최상위 작업 순서, 모듈 책임, 상용화 단계별 의존성을 한 파일에서 확인할 수 있도록 한다.

===============================================================================
1. 최상위 작업 순서
===============================================================================

00. 루트 계약 파일
    - package.json, pnpm-workspace.yaml, turbo.json, .env.example, README.md, SECURITY.md, MANIFEST.json, TREE.txt
    - 모노레포 실행 방식, workspace 범위, 환경변수, 보안 기준, 전체 구조를 확정한다.

01. docs/
    - 서비스 기획, PRD/BRD, UX/UI, 기능명세, 데이터, API, 기술설계, 보안, QA, 운영, 출시, 협업 문서를 확정한다.
    - 이후 database, packages, services, apps 구현의 기준 문서가 된다.

02. database/
    - ERD, migrations, policies, seeds, 백업 기준을 확정한다.
    - 급여·예산·지출·저축·알림·성장·커뮤니티·광고·관리자 운영 데이터의 서버 권위 원천이다.

03. packages/
    - api-contract, db, security, types, ui, utils, config 공통 모듈을 확정한다.
    - apps와 services가 같은 타입/계약/보안/유틸을 사용하게 한다.

04. services/
    - api, notifications, scheduler를 구현한다.
    - 서버 권위 계산, 인증/인가, 알림 발송, 월마감, 보존/삭제 cleanup을 담당한다.

05. apps/
    - mobile 앱과 admin 콘솔을 구현한다.
    - 사용자 급여관리 UX와 관리자 운영 UX를 제공한다.

06. assets/
    - 브랜드, 디자인, 프로토타입, 스토어 스크린샷 기준을 정리한다.

07. infra/
    - Cloudflare, Neon, domain, env, GitHub environment/secrets를 구성한다.

08. scripts/
    - DB, 개발환경, 문서생성, 품질검사, 릴리즈 자동화를 담당한다.

09. tools/
    - API contract check, DB schema check, doc lint, security audit 도구를 둔다.

10. .github/
    - issue template, PR template, CODEOWNERS, CI/CD workflow를 운영한다.

11. security/
    - threat model, privacy impact assessment, security checklist, vulnerability response를 관리한다.

12. observability/
    - logs, metrics, dashboards, alerts 기준을 관리한다.

13. qa/
    - API, 성능, 보안, test cases, UAT 인수검증을 담당한다.

14. ops/
    - 관리자 운영, 커뮤니티 모더레이션, CS, incident, 운영 지표를 관리한다.

15. release/
    - release notes, rollback, screenshots, store metadata, versioning을 관리한다.

===============================================================================
2. 최상위 폴더 책임 요약
===============================================================================

.github/        GitHub 협업, CI/CD, 보안 스캔, 배포, 릴리즈 자동화
apps/           사용자 모바일 앱과 관리자 콘솔
assets/         브랜드·디자인·프로토타입·스토어 자산
database/       PostgreSQL ERD, migration, policy, seed, 백업 기준
docs/           제품·기능·UX·API·기술·보안·QA·운영·출시 최종 문서
infra/          Cloudflare, Neon, domain, env, GitHub 인프라
observability/  로그, 메트릭, 대시보드, 알림
ops/            운영자 매뉴얼, CS, 모더레이션, 장애 대응
packages/       공통 타입, API 계약, DB, 보안, UI, 유틸, 설정
qa/             테스트 계획, 테스트 케이스, 성능/보안/UAT 검증
release/        출시, 롤백, 스토어, 버전 정책
scripts/        DB/dev/docs/quality/release 자동화
security/       위협모델, 개인정보 영향평가, 취약점 대응
services/       API, 알림 worker, scheduler
tools/          내부 검증 도구

4. 파일 단위 완료 기준
===============================================================================

이 TREE.txt는 다음 조건을 충족할 때 파일 단위 완료로 본다.

1. 최상위 작업 순서가 명확하다.
2. 루트 폴더와 루트 파일을 누락 없이 포함한다.
3. .github, apps, assets, database, docs, infra, observability, ops, packages, qa, release, scripts, security, services, tools 구조를 포함한다.
4. docs/final-documents 1~12번 최종 문서 세트를 포함한다.
5. database의 ERD, migrations, policies, seeds 구조를 포함한다.
6. services의 API, notifications, scheduler 구조를 포함한다.
7. apps의 mobile, admin 구조를 포함한다.
8. packages의 api-contract, config, db, security, types, ui, utils 구조를 포함한다.
9. QA, 보안, 운영, 출시 폴더를 상용화 단계까지 추적 가능하게 포함한다.
10. MANIFEST.json, TREE.txt, turbo.json 등 루트 계약 파일을 포함한다.
11. 미완료 표식, TODO, TBD, placeholder를 포함하지 않는다.
12. 실제 구현 순서와 폴더 책임이 서로 충돌하지 않는다.

5. 최종 판정
===============================================================================

이 TREE.txt는 급여납치 플랫폼의 최상위 폴더 구조, 하위 파일 구조, 구현 순서, 모듈 책임, 상용화 준비 구조를 통합 정리한 최종 파일 트리 문서다.

문서상·이론상 TREE.txt 파일 단위 기준으로 더 이상 추가 작성이 필요 없는 최종본으로 사용한다.

단, 프로젝트 종합 완성도 100%는 이 파일의 구조와 실제 repository 파일, package scripts, CI/CD, DB migration, API/RBAC/E2E/QA, 배포/운영 검증이 모두 일치할 때 최종 확정한다.
