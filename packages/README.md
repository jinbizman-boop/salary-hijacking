# packages

급여납치(Salary Hijacking) 플랫폼의 여러 앱, API, 워커, 관리자 콘솔, E2E 테스트가 공유하는 코드와 계약을 보관하는 워크스페이스 루트입니다. 이 폴더의 하위 구조는 급여납치 플랫폼 최종 파일트리 기준으로 고정하며, 급여·예산·지출·저축·알림·레벨업·커뮤니티·광고·운영 시스템이 동일한 타입, 유틸, UI 계약을 사용하도록 유지합니다.

## 역할

`packages`는 애플리케이션별 구현이 아니라 플랫폼 공통 기반을 제공하는 계층입니다. 서버 권위 계산, 클라이언트 표시, 관리자 운영, 자동화 테스트가 서로 다른 규칙을 갖지 않도록 공통 타입, 공통 유틸, 공통 UI, 공통 정책을 이곳에서 관리합니다.

## 고정 하위 구조

```txt
packages/
├─ types/      # 급여·사용자·지출·성장·알림·커뮤니티 도메인 타입 계약
├─ utils/      # 날짜, 금액, 페이지네이션, 문자열 공통 유틸
├─ ui/         # headless UI 컴포넌트와 디자인 토큰
└─ README.md   # packages 워크스페이스 운영 기준
```

## 패키지 책임

### `@salary-hijacking/types`

급여납치 플랫폼의 공통 타입 계약을 제공합니다. 급여 계획, 일일 예산, 고정지출, 고정저축, 변동지출, 알림, LV UP, 커뮤니티, 사용자/권한/동의/감사 계약은 이 패키지를 기준으로 정의합니다.

필수 원칙은 다음과 같습니다.

- KRW 금액은 1원 단위 정수로 표현합니다.
- 클라이언트는 최종 급여·예산·지출·저축 계산 권위를 갖지 않습니다.
- 사용자 토큰, 비밀번호, push token, 주민등록번호, 계좌번호, 카드번호, 원본 금융 데이터는 UI 또는 로그에 직접 노출하지 않습니다.
- 도메인 이벤트, idempotency key, audit snapshot을 통해 서버 쓰기와 월마감의 재현성을 보장합니다.

### `@salary-hijacking/utils`

앱, API, 워커, 관리자 콘솔, E2E가 함께 사용하는 순수 TypeScript 유틸을 제공합니다.

대표 모듈은 다음과 같습니다.

- `date`: Asia/Seoul 기준 급여일, 월 주기, 일일예산 기간, 고정지출/고정저축 예정일, 알림 예약, 감사 기준일
- `money`: KRW 정수 검증, 일일예산 사용/잔여/초과 계산, 납치금액 floor-zero 보조, 원화 포맷
- `pagination`: offset/cursor pagination, keyset 힌트, 도메인별 page size, 접근성 라벨
- `strings`: 문자열 정규화, 민감정보 redaction, 커뮤니티/알림/관리자 텍스트 sanitize, 검색어/hashtag 처리

### `@salary-hijacking/ui`

급여납치 화면을 구성하는 headless UI 컴포넌트와 디자인 토큰을 제공합니다. 앱 프레임워크에 직접 종속되지 않도록 표시 모델, 접근성 정보, 액션 메타데이터, policy guard를 반환하는 구조를 기본으로 합니다.

대표 영역은 다음과 같습니다.

- 급여 상태 카드
- 일일 예산 카드
- 지출 리스트 아이템
- 알림 행
- 커뮤니티 게시글 카드
- 색상, 간격, 타이포그래피 토큰

## 서버 권위 원칙

급여납치의 핵심 계산은 반드시 서버 권위로 처리합니다. `packages`는 타입과 보조 유틸을 제공하지만, 다음 작업의 최종 확정은 API/DB 트랜잭션 계층에서 수행해야 합니다.

- 급여 수령 확정
- 월별 납치금액 확정
- 월마감/재오픈
- 지출 생성/수정/취소/삭제 후 일일예산 재계산
- 고정지출/고정저축 반영
- 알림 발송 확정
- 광고/제휴 노출·클릭·전환 집계
- 관리자 제재/복구/감사 기록

## 품질 게이트

각 패키지는 최소한 다음 검증을 통과해야 합니다.

```bash
npm run verify
npm run quality
npm run typecheck
npm run build
```

공통 품질 기준은 다음과 같습니다.

- `strict` TypeScript 통과
- `exactOptionalPropertyTypes` 통과
- `noUncheckedIndexedAccess` 통과
- placeholder `export {}` 금지
- React/JSX 런타임이 필요 없는 패키지에서 React import 금지
- `dangerouslySetInnerHTML` 직접 사용 금지
- 민감정보 원문 로그/렌더링 금지
- completeness report 및 assertion 제공
- smoke test 또는 domain-level 검증 제공

## 보안 및 개인정보 정책

`packages` 계층은 공통적으로 다음 정책을 지켜야 합니다.

- 비밀번호, access token, refresh token, push token, secret은 원문 저장·노출·로그를 금지합니다.
- 주민등록번호, 계좌번호, 카드번호, 전화번호, 이메일은 필요한 경우 마스킹 또는 redaction합니다.
- 급여, 대출, 저축, 지출 원본 데이터는 광고 추천/타겟팅 데이터와 분리합니다.
- 민감 금융 데이터 기반 광고 타겟팅은 별도 동의 없이 수행하지 않습니다.
- 커뮤니티와 금융 원본 데이터를 직접 결합하지 않습니다.
- 관리자 기능은 role, permission, audit trail을 전제로 합니다.

## 도메인 반영 범위

`packages`는 급여납치 1~5단계 플랫폼 구축 범위 전체를 지원해야 합니다.

1. 기초 인프라, 모노레포, 경로/환경변수, 인증, 서버 권위 아키텍처
2. 급여/예산/지출/저축/알림/광고/운영 데이터 안정성
3. 급여 홈, 계획, 일일예산, 고정지출, 고정저축, 변동지출
4. 알림, LV UP, 커뮤니티, 글쓰기, 마이페이지, 관리자 콘솔, E2E/QA
5. 상용화 운영, 광고/제휴, 반응형 UI/UX, 성능 최적화, 배포 안정성

## 변경 규칙

- 공통 타입과 유틸 변경은 API, 앱, 관리자, 워커, E2E 영향 범위를 함께 검토합니다.
- 금액·날짜·권한·알림·광고·커뮤니티 정책 변경은 반드시 서버 권위 계층과 동기화합니다.
- 하위 패키지의 export map을 변경하면 import 경로와 build 산출물을 함께 검증합니다.
- 문서상 완료로 표시하려면 typecheck, build, completeness, smoke/e2e 기준을 모두 통과해야 합니다.

## 완료 기준

이 `packages` 루트는 다음 조건을 만족할 때 파일 단위 완료로 간주합니다.

- 하위 패키지의 책임과 경계가 명확히 문서화되어 있습니다.
- 서버 권위, 개인정보, 광고/커뮤니티 분리, 감사/멱등성 원칙이 명시되어 있습니다.
- 급여납치 핵심 기능 범위와 1~5단계 구축 기준을 지원합니다.
- 각 패키지 검증 명령과 품질 게이트가 명확합니다.
- 실제 레포 통합 시 `packages/types`, `packages/utils`, `packages/ui`가 동일한 계약으로 소비될 수 있습니다.
